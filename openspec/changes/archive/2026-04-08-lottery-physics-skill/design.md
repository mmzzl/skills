## Context

本项目将已有的 `physics_lottery` 物理引擎封装为 opencode skill。目标是让用户可以通过自然语言调用物理摇号模拟器，无需了解底层物理引擎的实现细节。

当前状态：
- `physics_lottery/` 模块已实现，包含完整的3D物理引擎和摇号逻辑
- 模块可通过 CLI (`python -m physics_lottery.main`) 或 Python API 调用
- 缺少 skill 封装，无法被 opencode 作为技能直接调用

约束：
- skill 必须遵循 opencode 的 skill 格式规范
- 需要支持自然语言参数解析
- 保持物理引擎的独立性和可测试性

## Goals / Non-Goals

**Goals:**
- 创建符合 opencode 规范的 skill 文件 (SKILL.md)
- 支持自然语言参数：前区范围+数量，后区范围+数量
- 支持 seed 参数实现可复现结果
- 提供清晰的输出格式

**Non-Goals:**
- 不修改物理引擎核心代码
- 不实现可视化或 Web 接口
- 不支持配置气流参数（保持简单）

## Decisions

### 1. Skill 调用方式

选择通过 Python subprocess 调用物理引擎模块：
- 优点：保持引擎独立，不增加依赖；输出格式化简单
- 缺点：性能略低（进程启动开销）

替代方案：直接在 skill 中导入并调用 Python 模块
- 优点：无进程开销
- 缺点：需要处理参数解析和输出格式化，增加复杂度

### 2. 参数解析方式

使用正则表达式解析自然语言参数：
```
前区(\d+)-(\d+)选(\d+)|后区(\d+)-(\d+)选(\d+)
```

示例输入："前区1-35选5，后区1-12选2"

### 3. 输出格式

```
🎱 摇号结果
前区: [3, 7, 15, 23, 31]
后区: [2, 9]
```

## Risks / Trade-offs

| Risk | Impact | Mitigation |
|------|--------|------------|
| 参数解析失败 | 无法执行摇号 | 返回错误提示，说明正确格式 |
| 物理引擎超时 | 长时间无结果 | 设置 CLI 超时（如60秒），超时则报错 |
| seed 冲突 | 多次调用结果相同 | 使用时间戳或 UUID 作为默认 seed |
