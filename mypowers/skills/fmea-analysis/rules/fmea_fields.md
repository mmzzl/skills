# FMEA 字段规范

|分组|字段|必填|AI填写/审核规则|
|---|---|---|---|
|功能分析|功能点/业务|是|必须拆到可验证的独立功能或业务，避免只写模块名。|
|功能分析|功能要求|是|写客户层面和系统层面的成功判定，包含时延、状态、数据一致性、可见性等可验收条件。|
|失效分析|失效模式|是|描述系统层面不希望出现的现象，如失败、超时、卡死、数据错误、性能下降。|
|失效分析|失效影响|是|重点写对上一层业务或最终客户业务的影响，而不是只重复功能本身失败。|
|失效分析|失效原因|是|写直接原因/故障模式；一个原因一行，避免把多个根因塞进一条。|
|风险分析|严重程度（S）|是|按 S评价准则 1~10 打分，越大表示影响越严重。|
|风险分析|发生概率（O）|是|按 O评价准则 1~10 打分，越大表示原因发生概率越高。|
|风险分析|检测度（D）|是|按 D评价准则 1~10 打分，越大表示越难检测；1 表示几乎肯定能检测或已有检测机制。|
|风险分析|AP（措施优先级）|是|必须由 S/O/D 按 AP评估表推导，不能凭感觉填写。|
|技术改进|改进措施|是|H/M 必须给措施；优先消除失效原因，无法消除时补检测定位、上报、隔离、恢复或人工修复方案。|
|技术改进|改进效果|否|写措施落地后的具体效果，尽量用可验证结果描述。|
|技术改进|责任部门/人|否|明确责任人或团队。|
|技术改进|完成时间|否|只填改进措施完成时间，不包含测试完成时间。|
|技术改进|完成状态|否|可填未开始/进行中/已完成/暂缓，并说明原因。|
|改进落地|重要程度|否|用于设计补充重要程度；BVT/Level1 可进可靠性基线，Level2/Level3 有测试用例覆盖但不一定进基线。|
|改进落地|开发自检|否|开发提测前检视是否覆盖、是否自验。|
|改进落地|用例ID|否|测试提测前检视是否有用例覆盖，并记录用例 ID。|
