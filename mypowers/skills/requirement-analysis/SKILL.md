# 需求分析

**Skill 标识**: `requirement-analysis`

其他 skill 通过 `requirement-analysis` 引用本 skill。

## 概述

将原始产品需求转化为结构化、可追溯的 AI 需求文档，遵循 Epic → Feature → Story 层次结构。本 skill 弥合了非正式用户需求与正式开发输入之间的差距。

**核心原则：** 以用户为中心的语言、可测试的验收标准、从业务价值到用户故事的完整可追溯性。

## 扩展点

开始之前，从插件根目录（此 skill 基目录向上 2 级——即 skill 加载时显示的"此 skill 的基目录"）读取 `cospowers.config.json`。无需回退——配置文件始终包含有效默认值。

| 配置字段 | 用途 |
|---|---|
| `config.env.SPEC_DEVELOPER_SERVER_URL` | 远程知识库服务器 URL，用于知识发现 |
| `config.kb.skill` | 要调用的知识库 skill（null = 使用 kb-query 自动检测） |
| `config.kb.localPath` | 本地知识库目录，用于基于文件的查找 |
| `config.templates["ai-requirement"]` | TR1 用户需求规格说明书的输出模板文件路径（默认：`templates/user-requirement-template.md`） |
| `config.templates["ai-requirement"]` | TR1 用户需求规格说明书的输入 schema 参考——使用同一模板解析已匹配该结构的输入文档 |
| `config.evaluators["aireq"]` | 在步骤 14 调度的评估器 skill 名称；`false` = 跳过关卡 |

## 何时使用

- 用户提供原始功能描述、PRD 或口头需求
- 需要正式需求文档的企业或多服务项目
- 用户明确请求需求分解或分析
- 启动端到端团队项目工作流（在 system-requirement-analysis 之前）

**以下情况不要使用：**
- 需求已结构化（跳至 system-requirement-analysis 或 brainstorming）
- 小微项目（直接使用 brainstorming）

       > "我已将 AI 需求背景情报简报写入 `docs/agent-rules/1-ai-requirements/input/init-aireq-template.md`，请在 IDE 中打开填写。**§1 原始需求描述** 和 **§2 目标用户画像** 为必填，其余按需填写。填写完成后告知我。"
       等待确认，然后重新读取并应用映射表。
     - **如果文件存在但仍有占位符**（`*(在这里...)*`）：告知用户至少完成 §1 和 §2，然后等待。
     - **如果文件已填写**：应用上述映射表。
   - **如果用户提供文件路径或附上文档**（XMind、PRD，任何格式）：自动将其复制到 `docs/agent-rules/1-ai-requirements/input/`——不要请用户移动。该文件是**内容输入**，用于预填 init 模板，并告知用户补充、确认。
   - **如果用户内联提供内容**（需求描述、PRD 文本）：将其视为内容输入，用于预填 init 模板，并告知用户补充、确认。
   - **架构图（init 模板 §5）**：提取组件/模块名称，在步骤 2 中作为搜索词添加。

   评估每个章节的**信息质量**（而不仅仅是非空）。提交一份完整性报告。**在所有阻塞项解决之前不要继续。**

   | §8 价值规划结论 | 否 | 有价值主张和量化效果目标 | 可留空 |
   | §9 历史质量问题 | 否 | 有 P0/P1 问题描述和来源 | 可留空 |
   | §10 跨部门协作需求 | 否 | 有协作方产品线和功能点 | 可留空 |

| §12 非功能性补充约束 | 否 | 国际化/NFV 已填写或标注"不涉及" | 可留空 |

   **报告格式：**
   ```
   📋 Init 完整性校验结果
   ✅ §1 原始需求描述：[一句话概括]
   ✅ §2 目标用户画像：识别到 N 个角色
   ⚠️ §6 领域术语：未填写，后续将使用通用术语（非阻塞）
   🚫 §1 原始需求描述：信息过于简单——请补充说明
   ```

   如果有任何阻塞章节不充分：告知用户具体缺少什么，等待更新，重新验证。非阻塞差距：注明并继续。

   **⛔ 在所有阻塞项通过之前不要进入步骤 2。**

2. **查询产品知识库** —— 如果你的项目有产品知识库工具（例如 `kb-query`），调用它来理解现有系统边界和业务流程（参见下方"知识库发现"章节）。如果未配置知识库工具，跳过此步骤。
3. **探索项目上下文** —— 检查现有文档、代码库、领域知识；调用 `daedalus-knowledge` 搜索相关团队标准、编码规范和技术指南（从 `cospowers.config.json` 读取 `SPEC_DEVELOPER_SERVER_URL`；如果未配置，则回退到本地 `rules/` 文件）。
4. **读取模板** —— 从 `config.templates["ai-requirement"]`（默认：`templates/user-requirement-template.md`）加载输出模板并内化其结构（9 个主要章节、QS 检查清单、DFX 基线等）。输出文档必须遵循此模板——这是一份 TR1 用户需求规格说明书。使用 Epic/Feature/Story 层次结构（参见分解方法）作为内部思维框架来组织内容，但将最终输出写入 TR1 结构。
5. **读取原始需求** —— 阅读 `docs/agent-rules/1-ai-requirements/input/` 中（或对话中）的所有输入材料，形成初步理解；此时尚不开始分解。

   **TR1 文档检测**：在阅读每个输入文档时，检查其结构是否匹配步骤 4 中内化的 `user-requirement-template.md` schema（即包含"详细用户需求规格"、"客户验证结论"、"QS-xxx"标记等章节）。如果匹配——该输入已是 TR1 用户需求规格说明书：
   - 将 §1.1.x（根特性）直接映射到输出 TR1 §1.1.x，用任何补充材料增强
   - 继承客户验证（§客户验证结论）和 QS 合规数据（文档完成自检清单）
   - 继承 DFX 基线（§2.2），用补充材料的额外分析丰富之
   - 在输出的材料概要中标记来源 TR1 文档
   - 对于输入 TR1 与完整模板之间的差距（例如缺失的 DFX 维度、不完整的 QS 检查），向用户标记以进行补充
6. **澄清模糊需求** —— 在分解之前，一次一个问题地消除歧义（参见下方"需求澄清"章节）。直到范围、假设和关键术语明确之后才能继续。
7. **确认补充材料** —— 询问用户是否存在额外材料：历史测试用例、之前版本的设计、竞品/供应商调研或其他约束。如果无，直接继续。
8. **加载和分级材料** —— 如果存在补充材料：
   - 少量（1-2 份文档）：直接加载并继续
   - 大量（3+）：在使用前分级：
     - **权威级**：主要需求文档、已确认的客户输入——用作事实依据
     - **参考级**：历史设计、竞品调研——有判断地使用
     - **孤立级**：测试用例、测试工具——仅提取意图，绝不让结构或命名污染需求
   - 在分解前产出一行材料概要
9. **分解为 Epic/Feature/Story** —— 遵循 3 级层次结构，使用以用户为中心的语言
10. **提交分解结果供评审** —— 展示树形结构，获取用户确认
11. **撰写 requirements.md** —— 将完整的需求文档输出到 `docs/agent-rules/1-ai-requirements/output/YYYY-MM-DD-<project>-requirements.md`（如果目录不存在则创建）。

    **运行时兜底策略：**
    - 信息不足时标记 `[待确认]`
    - 遇到 `[AI可读性要求]` 标注"AI 禁止填写"的章节 → 保持章节结构，每个填写位置填入 `[待确认]`，不编造内容
    - 确实不适用的章节 → 写"不涉及"+"原因"
    - **禁止编造数字** — 模板例值只是格式示意
12. **快速预检自查** —— 在调度评估器子代理之前，进行快速结构扫描（不是完整质量评审——那是步骤 14 的工作）：
    - **Epic**：全部 8 个必需章节存在且非空（从【关键价值点】到【责任人】）
    - **Feature**：全部 6 个必需章节存在（【关联关键价值点】到【包含 Story】）
    - **Story**：【安全需求】不为空，【技术思路】="待开发负责人填写"，【工作量评估】="待评估"，🔴/🟡/🟢 标记存在
    - **修复任何明显缺失的章节**——不要花时间在质量细节上，留给评估器子代理
13. **⛔ [待确认] 解决关卡** —— 扫描全文所有 `[待确认]` 标记，逐条列出并请用户逐条填写或提供材料。AI 根据用户输入更新文档。**在所有 `[待确认]` 标记解决之前不得继续。** 全部解决后进入步骤 14。
14. **质量评估（阶段关卡）** —— 使用 `config.evaluators["aireq"]` 中的 skill 名称（默认：`aireq-evaluator`），通过 `skills/<evaluator>/agents/evaluator-dispatch-prompt.md` 调度一个隔离的质量评估子代理。执行完整的逐条逐规则扫描（REQ-AI-01~34）+ 加权评分。应用所有报告的关键/错误修复。如果等级 < B（< 80）：向用户提交问题列表，请他们根据需要补充上下文，然后从步骤 9 重新运行。直到等级 ≥ B 方可进入步骤 15。

    如果 `config.evaluators["aireq"]` 为 `false`，跳过此关卡，直接进入步骤 15。
15. **用户评审输出** —— 请用户评审通过评估的文档，确认所有内容符合预期。根据需要迭代。完成后进入步骤 16。
16. **⛔ 转换关卡 —— 交接前验证评估器**：
    - **验证**：`aireq-evaluator` 是否以等级 ≥ B 通过？或者 `config.evaluators["aireq"]` 为 `false`？
    - **如果等级 ≥ B 或评估器已跳过**：直接调用 `system-requirement-analysis`。
    - **如果等级 < B**：不要转换。返回步骤 9，返工需求，重新运行步骤 14。
    - **如果无法确认**：不要询问用户。使用 `skills/aireq-evaluator/agents/evaluator-dispatch-prompt.md`，以 `docs/agent-rules/1-ai-requirements/output/` 中的当前文档重新调度 `aireq-evaluator`。如果 ≥ B 则继续，< B 则返工。

## 知识库发现

**查询之前，通过读取 `cospowers.config.json` 确定适用的模式：**

```
1. 知识库查找（补充上下文，使用最佳可用来源）：
   - config.kb.skill 已设置且环境中该 skill 可用？→ 调用该 skill
   - 否则 kb-query 在环境中可用？→ 使用 kb-query
   - 否则 config.kb.localPath 存在？（默认：doc/kb/）→ Grep/Read 本地知识库文件
   - 否则 docs/kb/ 存在？→ Grep/Read docs/kb/ 文件
   - 否则 → 跳过知识库查找

2. 代码优先：始终直接在代码库上 Glob + Grep + Read
```

**知识库指引方向，代码验证真相。** 无论何种模式，始终对照实际代码验证发现（步骤 C 如下）。

**在需求分析阶段，仅查询三件事：产品边界、模块范围和现有业务流程。不要查询 API 细节或技术栈——这些属于后续阶段。**

### 步骤 A：产品级理解（定义边界）

| 模式 | 操作 |
|------|--------|
| **kb-skill / kb-query** | 调用配置的知识库 skill 获取产品概览、模块数量、范围 |
| **local-kb** | `ls <config.kb.localPath>` → 阅读概览/架构文件 |
| **代码优先** | Glob 入口点（`main.go`、`app.py`、`__init__.py`）；阅读 README 和顶层目录结构 |

目标：理解新需求属于哪个层次，使 Epic 分解与现有领域边界对齐。

### 步骤 B：模块级理解（发现现有业务流程）

| 模式 | 操作 |
|------|--------|
| **kb-skill / kb-query** | 按关键词查询相关模块；阅读受影响模块的业务流程 |
| **local-kb** | `Grep "keyword" <config.kb.localPath>` → 阅读匹配的文件 |
| **代码优先** | 在源文件中 Grep 领域术语；阅读关键 handler/service/router 文件 |

每个模块要找到的关键内容：
- 项目结构 / 模块概览
- 架构 / 组件关系
- 现有业务流程（对需求差距分析至关重要）

**在此阶段不要查询**：API 细节、排障文档、技术栈（留给后续阶段）。

### 步骤 C：代码验证

无论何种模式，始终对照实际代码验证：

1. 使用 **Glob** 查找知识库或 README 中提到的文件
2. 使用 **Grep** 搜索关键函数、类、接口定义
3. 使用 **Read** 验证知识库描述是否准确
4. **当知识库与代码不一致时，代码为准**——相应更新你的理解

### 使用结果

- **架构理解**为 Epic/Feature 分解提供信息——与现有子系统边界对齐
- **模块知识**识别哪些功能已存在（避免重复指定）
- **业务流程**揭示当前状态与目标状态之间的差距（聚焦需求于增量部分）

## 需求澄清

在分解需求之前，消除所有歧义。**直到范围、关键术语和假设明确之后才能开始 Epic/Feature/Story 分解。**

### 规则

- **一次一个问题**——不要在一条消息中问多个问题
- **优先使用选择题**——在可能的情况下比开放式问题更容易回答
- **够清楚就停**——一旦你能回答"这个做什么、谁使用、什么不在范围内"，就继续

### 澄清什么

**范围边界**——确认包含什么以及明确不包含什么：
> "这个需求是否包含 X，还是不在本次迭代范围内？"

**模糊术语**——指出可能有多重含义的术语：
> "当你说'用户'时，是指最终用户、管理员，还是两者都包括？"

**隐藏假设**——在假设变成设计决策之前，将隐式假设明确化：
> "我假设这将替换现有的 Y 流程，而非并行运行——这是正确的吗？"

**多重有效解释**——当一个需求可以有两种解读时，选出并确认：
> "这可能意味着 A（行为 X）或 B（行为 Y）——你指的是哪一个？"

### 反模式

- 不要问实现细节——这些属于 system-requirement-analysis
- 不要问性能数字或技术约束——这些属于 DFX
- 不要批量问多个问题——问一个，等待回答，再问下一个

## 补充材料

在阅读主要需求文档后，在开始分解之前总是询问用户：

> "除了主要需求文档，你是否有以下任何材料？
> - 历史测试用例（有助于识别现有验收场景）
> - 之前版本的设计或需求文档（有助于区分增量功能和新功能）
> - 竞品或供应商调研（有助于定义场景边界和差异化点）
> - 文档中未包含的其他约束或已确认的约定"

**如果没有补充材料**：直接进入分解。

**如果材料较少（1-2 份文档）**：加载并阅读它们，与主文档一起使用。

**如果材料较多（3+）**：在使用前分级。不要对所有文档一视同仁。

| 等级 | 典型来源 | 如何使用 |
|-------|----------------|------------|
| **权威级** | 主要需求文档、已签署的客户输入 | 事实依据——直接使用 |
| **参考级** | 历史设计、竞品分析、调研文档 | 用架构师判断力使用，应用前过滤 |
| **孤立级** | 测试用例、测试工具、测试专用命名 | 仅提取用户意图——绝不让命名或结构污染需求设计 |

**测试用例的隔离规则**：测试用例标题用自然语言描述用户意图——有助于扩展验收标准。但测试基础设施、命名约定和实现假设不得进入主要需求结构。

在需求文档顶部产出一行**材料概要**：
```
输入：[主文档名]（权威级） + [X 个历史测试用例]（意图参考） + [Y 份竞品文档]（参考级）
```

## 分解方法

> **注意**：下面的 Epic → Feature → Story 层次结构是**内部思维框架**，用于组织分析。实际输出文档遵循 TR1 用户需求规格说明书结构（参见输出模板章节）。使用此分解来识别和结构化内容，然后将内容填充到相应的 TR1 章节（§1.1.x 根特性 → feature → 需求 ID）。

### 3 级层次结构

```
Epic（业务主题 / 产品模块）
  → Feature（完整的用户价值功能场景）
    → Story（具体的用户操作）
```

| 层级 | 定义 | 关注点 | 验收视角 |
|-------|-----------|-------|----------------------|
| **Epic** | 业务主题 / 产品模块 | 业务价值 | 市场价值交付 |
| **Feature** | 完整的功能场景 | 用户场景 | 功能集完整性 |
| **Story** | 具体的用户操作 | 用户行为 | 用户验收 |

### Story 编写规则

每个 Story 必须遵循：

**用户故事格式：**
```
作为 <用户角色>，我希望 <功能/行为>，以便 <实现的价值>
```

- 必须从外部用户视角出发，而非内部系统角色
- 使用业务语言，非技术人员应能理解
- 每个 Story 应可独立测试

### 简化验收标准（A/C）

每个 Story 需要：

**正常场景**（必填）：1-2 个典型的成功路径场景
- **Given**：描述前置条件和初始状态（使用业务语言）
- **When**：描述具体的用户操作（避免技术细节）
- **Then**：描述用户可见的预期结果

**主要异常场景**（可选）：最常见的错误或异常情况

### 质量规则

1. 聚焦核心场景，不要求覆盖所有边界情况
2. 使用业务语言，非技术人员也应能理解
3. 每个 Story 的验收标准控制在 2-4 个场景
4. 标记需要深入分析的项目，留给系统需求阶段

## 输出模板

输出文档必须遵循 TR1 用户需求规格说明书模板。在步骤 4 中从 `config.templates["ai-requirement"]`（默认：`templates/user-requirement-template.md`）加载模板，并在撰写前内化完整结构。文档具有以下关键章节和要求：

**第 1 章 —— 详细用户需求规格**（含用户场景分析的根特性）：
- 每个 `§1.1.x 价值根特性` 代表一个业务价值领域（战场）
  - `§1.1.1 用户场景分析` —— 用户旅程、场景建模、痛点（量化、有来源）、价值规划、竞品分析
  - `§1.1.2 设计策略总结` —— 设计意图/目标（可量化）、设计策略、创新点
  - `§1.1.3 产品主场景 Playbook` —— 客户痛点 → 解决方案 → FAB 卖点
  - `§1.1.4 特性详细需求规格` —— 产品特性含 Playbook、使用场景、约束规格及遗留问题/关键劣势、产品特性列表（系统级主特性/子特性含价值构建策略和优先级）、关键遗留问题。
  - `§1.1.5 技术需求设计与评审结论` —— 可行性评估、预研项目、风险
- `§1.3 关键质量问题改进说明` —— P0/P1 质量改进及来源
- `§1.4 跨部门协作需求识别与共识确认` —— QS-002：跨团队协作需求及共识确认记录（共识确认记录表）

**客户验证结论** —— QS-001：≥3 个客户验证及证据。包含验证概要。此章节由人工填写（模板含 `[AI可读性要求]` "AI 禁止填写"），AI 在填写位置写入 `[待确认]`。

**质量需求** —— QS-003：P0/P1 问题池。列出问题及来源（遗留/混沌工程/客户问题）、影响范围以及是否纳入本版本。

**第 2 章 —— 非功能性需求**：
- `§2.1 产品规格汇总` —— 相对上版本的规格变化、硬件支持、许可
- `§2.2 DFX 需求基线` —— QS-004：完整 DFX 基线覆盖
  - 性能参数（L0 底线：Web 控制台高频页面 99% ≤3s）
  - 可靠性（SLA、故障检测、HA、危险操作防护）
  - 可用性（业务可用性指标）
  - 合规性（QS-007：道德合规 Checklist 4 个类别）
  - 开放性（标准化接口）
  - 兼容性（OS、浏览器、硬件、升级路径）
  - 可服务性（可交付性、可维护性）
- `§2.3 国际化支持` —— QS-006：海外基线自检（如不适用可跳过）
- `§2.4 NFV 支持` —— NFV 版本同步状态

**XAAS 客户成功相关需求** —— QS-008：客户成功需求（非 XAAS 可跳过）

**第 3 章 —— 附录** —— 术语表、变更历史

**文档完成自检清单** —— 提交前强制自检：
- 全部 8 个 QS 项已检查（QS-001~008），含通过/不通过及证据链接
- AI-Native 格式检查（frontmatter 含 `stage: TR1`、高密度概要、无模糊词、所有 `[待确认]` 标记已列出）

**关键规则：**
- **禁止模糊词**：合理 / 适当 / 较好 / 尽量 / 性能好 / 适当考虑 —— 严格禁止
- **客户验证 ≥3**（QS-001，不可协商）
- **产品规格基线 + DFX 基线全覆盖**（QS-004，不可协商）；不达标的项目必须有 Story 跟踪或自检结论
- **安全需求明确声明**（QS-007）——"不涉及"可接受，空白不可接受
- **所有占位文本** `*[说明文字]*` 必须替换为实际内容
- **含** `[AI可读性要求]` "AI 禁止填写"**的章节** —— AI 不编造内容，在填写位置写入 `[待确认]`
- AI 不分配人员、不估算工时、不创造技术设计——这些标记为"待填写" / "待评估"
