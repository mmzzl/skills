# 设计核心哲学调研报告

---

## 一、设计的第一性原理

> "我认为设计的'第一性原则'，就是以最高的效率消除复杂性带来的风险。"

**推导逻辑：**
1. 为什么需要设计？因为项目复杂度已经大到"不先想清楚，写不了代码"的程度，大到"不写文档，根本就想不清楚"的程度。
2. 导致我们需要做设计的根本原因，就是**复杂性**。
3. 设计的目的，就是**消除这些复杂性带来的风险**。
4. 评判标准：一个设计活动是否有效，看它"对消除风险有什么帮助"，再看用的方法是不是"最高效的方法"。

**操作框架 "风险驱动的设计方法":** "一条基本原则，两个分析过程，三种纠偏活动"

| 要素 | 说明 |
|------|------|
| 一条基本原则 | "需求是我们的靶心，是我们要命中的目标；风险是我们的准星，是我们能瞄准的关键" |
| 两个分析过程 | 需求分析过程（梳理清楚到底需要做什么）、风险分析过程（确认是否达成目标，是否遗留风险） |
| 三种纠偏手段 | 沟通、检视、评审 |

## 二、设计三步法

| 步骤 | 名称 | 要点 | 典型产出 |
|------|------|------|----------|
| 第一步 | **确定目标** | 需求梳理，风险预估 | 需求TODOLIST，标注好风险级别的需求项 |
| 第二步 | **确定方案** | 方案构思，风险分析 | 方案思路、三视图（结构/逻辑/数据）、风险应对措施 |
| 第三步 | **确定细节** | 逐步细化，查缺补漏 | 模块定义列表（总设）/ 函数定义列表（概设） |

**完成标志 [文档直接说的]:**
- 第二步完成标志：中高级别的风险已经消除，不确定性已经消除
- 第三步完成标志：可以准确估算工作量（代码行数和开发工作量）

**粒度标准：**
- 总设输出的模块粒度：一个人一个版本周期内可完成
- 概设输出的函数规模：控制在50行代码以内

## 三、设计能力成熟度模型

| 层级 | 名称 | 标志 |
|------|------|------|
| **L1** | 无能力 | 完全不清楚该怎么去做，或者无法评估做出来的事情的好坏 |
| **L2** | 偶尔成功 | 掌握了部分方法但不熟悉，只能在部分场景下运用，需资深人员指导 |
| **L3** | 可重复 | 掌握了基本方法，能够熟练应用，大部分场景都能正确应用，可独立完成 |
| **L4** | 精确定义 | 熟能生巧，每次结果达到基本相似效果，统计数据平稳，能准确评判结果好坏 |
| **L5** | 优化级 | 能创新、改良方法，能针对该领域新问题给出有效解决方案 |

> "一次成功不叫成熟。次次都能成功，而且成绩稳定在一个合理范围之内，才叫成熟。"

## 四、设计质量评价体系

### 七大评价要素及默认权重

| 评价要素 | 默认权重 | A级(90+)标准 |
|----------|----------|-------------|
| **设计目标达成** | **40%** | 所有需求点描述完整，任意编码人员按设计都能良好实现 |
| **风格/可读性/文档规范性** | **10%** | 无需作者讲解，相关领域人员也能理解核心设计意图 |
| **易实现性/可维护性** | 10% | 模块化程度高，易分工，修改局部不影响其他部分 |
| **可调试/可测试/可运维性** | 10% | 模块可单独验证，能自动报告问题引入点 |
| **安全性** | 10% | 任何操作都不带来数据泄露/破坏/不可用 |
| **鲁棒性/健壮性/可靠性** | 10% | 各类异常场景都有分析设计，自动发现及恢复 |
| **可扩展/可复用性** | 10% | 有合理扩展机制，扩展不影响现有逻辑 |

### 设计红线（触发直接定级为C）

| 编号 | 红线类型 |
|------|----------|
| 1 | 可靠性章节缺失且无解释说明 |
| 2 | 安全性章节缺失且无解释说明 |
| 3 | "设计目标达成"或"风格/可读性"任一为C |
| 4 | 未使用最新模板且缺失章节>=3个 |
| 5 | 未进行设计checklist自检 |

## 七、发现的矛盾

1. **风险公式版本差异**: 1.2.1 的风险值 = 出问题概率×影响程度×影响广度（3维度），1.4.2 的风险级别 = 重要程度×实现难度×变更可能性×影响面（4维度）。两者面向不同阶段（需求风险 vs 确定目标时的优先级），不是矛盾，但术语不统一。
2. **文件命名冲突**: 存在 `1.4.4_设计质量评估` 和 `1.4.4_设计质量评价` 两个目录，可能是同一文档两个版本。
