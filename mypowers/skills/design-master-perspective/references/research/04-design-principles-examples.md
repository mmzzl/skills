# 设计原则与范例调研报告

---

## 一、模块设计规范 (1.9)

**模块的标准组成：**
目录结构、接口、配置、日志、通讯、存储、文档、单元测试、一键发布、基础支持

**模块特性要求：**
模块化、可调试性、可测试性、可靠性、安全性、可复用性、可扩展性、可运维要求

---

## 二、软件设计七大原则（来自1.7方法论的系统提炼）

**已有调研确认的设计原则：**

| 原则 | 来源 | 核心含义 |
|------|------|---------|
| 单一职责原则 | 1.5.4, 1.7.7 | 一个模块/函数只做一件事 |
| 开闭原则 | 1.6.6, 1.7.7 | 对扩展开放，对修改关闭 |
| 最少知识原则 | 1.7.6.2 | 模块间尽可能少相互依赖 |
| 高内聚低耦合 | 1.5.4, 1.7.7 | 模块内联系 > 模块间联系 |
| 机制策略分离 | 1.5.6, 1.5.8 | 稳定的机制层 + 变化的策略层 |
| 边界原则 | 1.5.4 | 改变的影响不会超出边界 |
| 不要瞎掺和 | 1.5.8 | API 设计不越权处理上层业务 |

**UNIX设计哲学（来自已读文档的体现）：**
- 一切皆文件（统一抽象）
- 管道机制（组合优于继承）
- 简单优先（Do One Thing Well）

## 三、设计中的一致性

在已读文档中反复体现：
- API命名规范一致（1.5.8）
- 三视图方法中每一层视图内部一致
- 七大评价要素中"风格/可读性/文档规范性"作为10%固定权重
- 命名练习（1.14.7）反复强调同一概念必须用同一名词

## 四、范例的核心教训（从已读内容提炼）

**API重构范例核心教训（sf_license API）：**
- v1：数据结构暴露在头文件中（破坏封装）
- v4（好设计）：通过不透明句柄 + accessor 函数隐藏实现，调用者只知接口不知内部

**好设计 vs 坏设计的通用判断标准：**
1. 好设计：改变内部实现不需要修改调用者（封装）
2. 好设计：接口语义清晰，见名知义（命名）
3. 好设计：可以独立测试，不依赖外部环境（可测试性）
4. 好设计：易变的部分和不变的部分分离（解耦）
5. 坏设计：一处改动导致多处受影响（高耦合）
6. 坏设计：抽象层过多导致追踪困难（过度设计）

---

## 五、典型问题目录（真实项目中收集）

7类练习题均来自真实项目：
1. 过度耦合 — 逻辑与I/O强耦合，无法单元测试
2. 过度设计 — 不需要扩展却引入了抽象层
3. 冗余代码 — 相似逻辑多处复制粘贴
4. 滥用机制 — 不当使用C++类/宏/模板
5. 异常处理 — 错误处理层次错误或被吞掉
6. 界面缺失 — 模块无清晰公共接口
7. 命名问题 — 混淆/不完整/误导/作用域污染
