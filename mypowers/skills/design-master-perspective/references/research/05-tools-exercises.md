# 设计原则与反模式调研报告

## 一、设计工具

### 具象化思考核心论点

> "文不如表，表不如图"

**图形的两大作用：**
1. **利于表达** — 人脑（爬行脑）天生偏好视觉化信息，图形是二维的，信息密度远高于文字
2. **利于思考** — 将思考过程具象为图形，可"外化记忆"，每次只聚焦一小部分关系

**三种工具层级：**
```
图形（降低认知负荷，利于表达关系）
  |
表格（精确数据对比、核对跟踪）
  |
文字（最终沟通存档）
```

## 二、设计反模式（七大维度）

| 维度 | 核心问题 | 危害 | 修复方向 |
|------|---------|------|---------|
| **过度耦合** | 逻辑与执行环境强绑定，无法注入 | 不可自动化测试，需真实环境才能运行 | 外化依赖，提纯逻辑层 |
| **过度设计** | 不必要的抽象/间接调用 | 可读性下降，追踪困难，维护成本高 | 保持简单直接，抽象要有扩展理由 |
| **冗余代码** | 相似逻辑多处重复 | 修改时需多处同步，一致性风险高 | 提取公共函数，消除重复 |
| **滥用机制** | 不当使用语言/框架机制 | 复杂化代码，超出机制设计意图 | 按场景选择合适机制 |
| **异常处理不当** | 吞异常/过宽捕获/错误层次错误 | 异常信息丢失，错误传播不可控 | 在正确层次处理，下层检测上层处理 |
| **界面缺失** | 无清晰接口抽象，直接操作实现 | 模块边界模糊，耦合度高 | 定义显式接口，封装内部实现 |
| **命名问题** | 混淆/不完整/误导/作用域冲突 | 阅读者误解代码意图，隐藏 bug | 名字要反映完整语义，加作用域前缀区分 |

### 命名四类典型问题（来自真实范例）：
1. **功能歧义** — 统计宏和处理函数名字几乎相同，无法区分职责
2. **遗漏语义** — 宏名只体现"检查"，遗漏了"获取头部"这一关键副作用
3. **语义反转** — `DBG_SIP` 被误读为"开启调试"，实际是"匹配条件"
4. **作用域污染** — 同名 `fd` 出现在全局/局部/宏参数三个作用域，指向不同文件

## 三、知识体系映射

七堂课主题 vs 七大反模式：

| 七堂课主题 | 对应反模式 |
|-----------|----------|
| 可测试性 | 过度耦合（无法测试） |
| 模块性 | 界面缺失（无清晰边界） |
| 设计风格 | 命名、冗余代码 |
| 设计模式 | 滥用机制（模式误用） |
| 可扩展与可复用 | 过度设计（为扩展而扩展） |
