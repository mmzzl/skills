# 功能性与非功能性设计调研报告

---

## 一、功能性设计核心方法

### 1.1 需求分析 [1.5.1]

> "设计阶段做需求分析的最主要目的，不是搞清楚用户需要什么，而是搞清楚设计需要做什么，怎么更好的实现用户'真实'的需求，消除实现过程的技术风险。"

**先发散后收敛：**

**发散（4个固定维度）：** 用户需求、场景（角色+条件+动作+事件）、外部接口、设计需求（可调试性/可测试性/可扩展性等）

**收敛（4个分析维度）：** 重要性、复杂度、影响面、风险

**挑战需求的三个问题：**
1. 这个需求真正要解决什么问题？
2. 如果这个问题不解决会怎样？
3. 这个需求是最好的解法吗？

> "所有你认为的需求，其实都是别人想的解决方案。"

---

### 1.2 方案设计 [1.5.2]

**五步：**
1. 工作原理 — 用用户可理解的语言描述设计逻辑
2. 场景分析 — use case 或场景设计法
3. 方案选型 — 评分法或对比法，关键：**当选型依赖的评估准则发生改变时，需重新评估方案**
4. 三视图 — 结构视图（模块静态）+ 数据视图（典型场景数据流）+ 逻辑视图（API实现工作流）
5. 开发计划 — 先框架，再核心逻辑，TDD验证

---

### 1.3 方案风险分析 [1.5.3]

> "就像赛车手上路比赛前，都会先上赛道跑上一遍，做到所有路况烂熟于胸。"

- 需求阶段风险分析：推测性的，识别可能风险，分配时间/精力
- 方案阶段风险分析：确认性的，基于确定设计，脑内模拟开发过程，找技术难点，准备预案

---

### 1.4 模块结构设计 [1.5.4]

**三种结构设计视角：**
1. 以数据结构为核心 — 数学模型/算法驱动
2. 以对象模型为核心 — 概念丰富、关系复杂时用类图
3. 以业务逻辑为核心 — 流程驱动，用数据流图

> "结构设计的好坏，标志就是'高内聚、低耦合'"
> "模块设计中必须打磨一个关键的概念——'边界'，即改变的影响不会超出边界。"

**模块分解的5种方法：** 按逻辑、按时间、按流程、按通信、按功能

**关键原则：** 按层序遍历金字塔（先广度后深度），"纵向切分业务，横向划分层次"

---

### 1.5 流程设计 [1.5.5]

**5种流程图选择：**
| 问题特征 | 图示 |
|---------|------|
| 控制流变化 | 标准流程图 |
| 数据流变化 | 数据流图 |
| 对象状态变化 | 状态图 |
| 对象协作 | 时序图 |
| 算法/数据结构逻辑 | 自定义示意图 |

**流程设计三要点：** 输入定义、输出定义、检查点（可能出错的地方）

---

### 1.6 数据结构设计 [1.5.6]

**四个评价标准（存在内在冲突）：**

| 标准 | 说明 |
|------|------|
| **高效率** | 快速操作，低资源消耗。不同效率指标相互冲突（速度 vs 资源，吞吐 vs 响应时间） |
| **强容错** | 难以误用，错误易检测和定位。文本格式比二进制格式容错性强 |
| **易使用** | 公共数据结构（RPC）要求高于私有数据结构（内存） |
| **易扩展** | 新内容不破坏既有逻辑 |

> "数据结构设计，首先得把需求（场景、目标）想清楚，并且，从中做出选择，选择我们要什么，**更要选择我们不要什么**。"

**机制与业务数据分离原则：** 机制层提供承载而不理解业务数据含义，是解耦的一种形式。

---

### 1.7 通讯协议设计 [1.5.7]

> "一般来说，我们应当选择更为简单的通讯模型。一对一模型能够解决问题的，就不要用一对多、多对多。"

**强烈建议：** 使用 Thrift / gRPC / RESTful API + PROTOBUF，而非自定义UDP、二进制格式或裸协议。

---

### 1.8 API 设计 [1.5.8]

> "API设计是最重要的一环...是决定整个项目质量最关键的一步"
> "不要瞎掺和应该成为API设计的基本原则"

**API设计三大原则：**

| 原则 | 说明 |
|------|------|
| **简约** | 数据结构最小化，参数用简单基础类型，避免手动资源管理 |
| **一致性** | 命名规范一致，返回值规范一致，行为约定一致 |
| **低耦合** | "不要瞎掺和" -- 通过回调/钩子解耦上下层，不让基础层理解业务层 |

> "没有文档的API就是耍流氓"（文档中重复强调30次）

**C++ 警告：**
> "在分层的模块开发中，C++并不适合用于机制（相对底层的API）的开发。" 原因：C++ 类必须在头文件中（相当于公开结构体），增加私有成员导致 ABI 破坏。

## 二、非功能性设计（DFX）

### 2.1 可靠性设计概念辨析 [1.6.1]

| 概念 | 定义 |
|------|------|
| 可靠性 | 在规定条件和时间内完成规定功能的能力 |
| 健壮性 | 在干扰因素存在时维持预期功能的能力 |
| 安全性 | 防止机密数据泄露给未授权用户 |
| 稳定性 | 在一定负载下长时间稳定运行的能力 |
| 可用性 | 可靠性+可维护性+保障性的综合体 |

> "高可用是可靠性的极致体现，而且，要达成高可用，只能通过架构上的设计来奠定基础。"

---

### 2.2 需求级可靠性 [1.6.1.1]

**SLA维度：**
- **服务可用性（"几个9"）：** 99.999% = 每年最多5分钟宕机，要求双路冗余+5分钟内自动切换
- **RTO（恢复时间目标）：** 极短RTO意味着需要集群或热备
- **RPO（恢复点目标）：** RPO越短，写磁盘频率越高，性能越差

> "设计系统，不能有侥幸心理...要当最坏的情况总会发生去考虑设计，需要通过机制保证最坏情况发生时有能力处理应对"

---

### 2.3 模块级可靠性 [1.6.1.3]

**异常处理策略（A-J）：**

| 策略 | 描述 | 适用 |
|------|------|------|
| A | 忽略错误继续 | 极少场景 |
| B | DEBUG only assert/abort | 仅程序员引发的异常 |
| C | 打印错误退出 | 小工具 |
| D | 重试 | 网络错误 |
| E | 丢弃坏输入继续 | 解析器、通讯协议 |
| F | 报警+停止等待人工干预 | 交互程序 |
| G | 自重启 | 资源耗尽 |
| H | 关闭受影响功能，降级继续 | 服务 |
| I | 尝试备选方案 | 服务 |
| J | 使用预设默认值 | 服务 |

**规则：** 系统引发的异常，绝对不使用A或B。

> "有个20/80法则...花在实现正常功能上的时间往往只有20%，花在排除各类异常所造成错误的时间是80%。"

---

### 2.4 函数级可靠性 [1.6.1.4]

**四个可靠性保证层级：**

1. **绝对成功保证** — 函数永不失败，或有回退措施
2. **事务级别** — 要么完全成功，要么零影响（对长期运行的服务，每个函数都应达到此级）
3. **系统可恢复** — 无法精确恢复但调用者可检测并恢复到已知状态
4. **错误不可知** — **这是一个bug，任何程序都不应该出现这种情况**

**错误处理的三条契约：**
1. 下层函数检测错误，上层函数处理错误（下层缺少上下文）
2. 函数必须清理自己造成的混乱（分配的内存必须释放）
3. **在最早可以检测出错误的地方进行检测，并截断错误的传播**

---

### 2.5 可扩展性设计 [1.6.6]

> "可扩展性，其实就是程序的套路（routines/patterns）。"

**设计视野层级：**
- 普通架构师：考虑当前需求
- 好的架构师：考虑2-3年需求
- 优秀架构师：考虑3-5年需求

**特殊处理是毒药：**
> "当一段机制代码需要根据上层逻辑进行特殊处理，就意味着机制无法满足需求了，这个时候应该重新整理机制，而不是加一段if语句进行特殊处理，所有的特殊处理都将导致架构腐烂。"

---

### 2.6 DDIA的可靠性、可维护性、可伸缩性 [1.6.7]

**三大核心关注点：**

**可靠性：**
- 故障（Fault）vs 失效（Failure）：组件偏离规格=故障，系统停止服务=失效
- 主动触发故障（Chaos Monkey）以验证容错机制
- 人为错误是宕机首要原因（占10-25%是硬件），不是硬件错误

**可伸缩性：**
- 用负载参数描述（请求/秒、读写比、活跃用户数）
- **百分位比均值更重要：** p50, p95, p99, p999 — 尾部延迟直接影响用户体验
- Amazon：响应时间增加100ms，销售量减少1%

**可维护性三原则：**
1. **可操作性** — 监控可见、自动化支持、良好文档
2. **简单性** — 消除**意外复杂度**（非问题本身固有的），最有力工具是**抽象**
3. **可演化性** — 与简单性和抽象密切相关

## 三、反复出现的跨功能主题

| 主题 | 体现 |
|------|------|
