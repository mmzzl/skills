# 系统需求分析

**Skill 标识**: `system-requirement-analysis`

其他 skill 通过 `system-requirement-analysis` 引用本 skill。

## 概述

将 AI 需求深化为遵循团队标准模板结构的正式系统需求文档。输出为一个包含 `index.md` 与 9 个章节文件的系统需求规格说明书，章节与模板保持一致：文档说明、关键应用场景、功能性需求（REQ-XXX）、DFX 非功能性需求、关键测试点、平台约束需求、设计任务书、需求优先级说明和修订记录。每个功能性需求项需包含完整描述、正常/异常/安全/边界/可测试性场景、依赖与风险、当前实现及问题、技术核实结论，并使用可客观判定的验收标准。

**核心原则：** 同一输出目录下的结构化多章节文档集，`index.md` 与 9 个章节文件完整填写（如不相关则注明"不适用"及理由），需求独立可验证，所有场景以具体可测试标准覆盖。

## 扩展点

开始之前，从插件根目录（此 skill 基目录向上 2 级——即 skill 加载时显示的"此 skill 的基目录"）读取 `cospowers.config.json`。无需回退——配置文件始终包含有效默认值。

| 配置字段 | 用途 |
|---|---|
| `config.env.SPEC_DEVELOPER_SERVER_URL` | 远程知识库服务器 URL，用于知识发现 |
| `config.kb.skill` | 要调用的知识库 skill（null = 使用 kb-query 自动检测） |
| `config.kb.localPath` | 本地知识库目录，用于基于文件的查找 |
| `config.templates["system-requirement"]` | 系统需求文档的模板文件路径 |
| `config.evaluators["sysreq"]` | 在步骤 4.2 调度的评估器 skill 名称；`false` = 跳过关卡 |

## 何时使用

- **路线 A**：AI 需求（Epic/Feature/Story）已存在且需要系统级深化（从 `requirement-analysis` 调用）
- **路线 B**：结构化用户需求文档已存在——跳过 `requirement-analysis`，直接从此处开始（从 `brainstorming` 场景 B 调用）
- 需要正式系统需求文档的企业项目
- 用户明确请求生成系统需求文档（例如，"生成系统需求文档"）

**以下情况不要使用：**
- 原始想法尚无任何文档（请先使用 `requirement-analysis`）
- 小微项目（直接使用 brainstorming）

```!
node ${CLAUDE_SKILL_DIR}/scripts/read-skill-supplement.mjs system-requirement-analysis ${CLAUDE_PLUGIN_ROOT}
```

## 工作流

你必须为每个主要步骤（0.1 到 4.2）创建一个 TodoWrite 任务，并按顺序完成。

### 步骤 0：背景情报与准备

**⛔ 关卡 —— 在步骤 0 完全完成之前不要进入步骤 1。**

#### 0.1 Init 模板

**Init 模板章节 → 分析步骤映射：**

| 章节 | 提取内容 | 输入到 |
|---|---|---|
| §1 业务背景 | 业务上下文和问题描述 | 步骤 1：为 REQ 项描述和输出文档第 1 章提供信息 |
| §2 关键业务场景 | 用户场景（自由文本） | 步骤 1：基于场景的 REQ 项的主要输入 |
| §3 历史版本文档 | 之前需求/设计文档的文件路径 | 步骤 0.5：加载以进行变更差异分析；取代询问用户 |
| §4 关联服务和系统 | 服务/模块名称 | 步骤 0.6：知识库/架构分析的主要搜索词 |
| §5 现有架构图 | 图表路径、draw.io 文件、Mermaid 代码或链接 | 步骤 0.6：提取组件名称 → 添加到知识库搜索；第 6 章平台约束 |
| §6 领域术语和业务规则 | 领域术语和业务规则 | 步骤 1：第 1 章定义；为 DFX 约束提供信息 |
| §7 其他上下文 | 自由文本额外上下文 | 应用于所有步骤的背景知识 |
| §10 结构化补充信息 | 手动填写的数据模型字段、API接口、枚举/状态机、业务规则 | §1–§2 确认填写后：读取 §10.1–§10.4 四个子节，将提取结果作为步骤 1 REQ 生成的结构化补充输入 |

**模板解析逻辑：**

**⛔ 重要：模板写入时的知识库预查询。** 当从用户输入或对话上下文中可识别模块/服务名称时，在**写入**模板之前使用这些名称调用知识库 skill（按 `config.kb.skill` 或 `kb-query`）。用结果预填 §9 DFX 约束并补充 §4——用户在首次打开时看到的是丰富的内容。步骤 0.6 将使用来自已验证模板的确认名称进行更深入的跟进。

- **如果在角色上下文中运行**（`requirement_analyst` 或 `ainative_requirement_analyst`）：检查 `docs/agent-rules/2-system-requirements/input/init-sysreq-template.md` 是否存在预装文件。
  - **如果文件不存在**：读取 `templates/init-sysreq-template.md`，按上述规则调用知识库预查询，然后将模板（如已提及项目名则预填，§9 DFX 基线和 §4 用知识库结果填充）写入该路径。然后告知用户："我已将背景情报简报写入 `docs/agent-rules/2-system-requirements/input/init-sysreq-template.md`，请在 IDE 中打开并填写。**§1 业务背景** 和 **§2 关键业务场景** 为必填项，其余按需填写，不知道的跳过。填写完成后告知我。"等待用户确认，然后重新读取并应用映射表。
  - **如果文件存在但仍有占位符**（`*(在这里写...)*` 或 `*(在这里贴`）：告知用户："请先在 IDE 中填写 `docs/agent-rules/2-system-requirements/input/init-sysreq-template.md` 的背景情报简报（§1 业务背景和 §2 关键场景为必填），完成后告知我。"等待用户确认，然后重新读取并应用映射表。
  - **如果文件存在且已填写**：应用上述映射表提取所有背景知识。
- **如果没有角色上下文且无文件提供**：将 init 模板写入项目并等待：
  1. 从 skill 的基目录读取 `templates/init-sysreq-template.md`
  2. 预填任何已知信息（如已提及项目名，对话中已可见的相关服务；按上述规则调用知识库预查询以填充 §9 和 §4）
  3. 写入 `docs/agent-rules/2-system-requirements/input/init-sysreq-template.md`（如需则创建目录）
  4. 告知用户："我已将背景情报简报写入 `docs/agent-rules/2-system-requirements/input/init-sysreq-template.md`，请在 IDE 中打开并填写。**§1 业务背景** 和 **§2 关键业务场景** 为必填项，其余按需填写，不知道的跳过。填写完成后告知我，我再开始分析。"
  5. 等待用户确认，然后读取文件并应用映射表。如果 §1 和 §2 仍为空，请用户仅完成这两个章节。
- **如果用户提供文件路径**（XMind、用户需求文档、PRD）：自动将其复制到 `docs/agent-rules/2-system-requirements/input/`——不要请用户移动。该文件是**内容输入**。Init 模板仍然**必需**——它收集文档中没有的背景上下文（业务动机、相关服务、领域术语）。检查 init-sysreq-template.md 是否存在并已填写；如果没有，生成它（从文档预填已知信息）并请用户在继续前完成 §1 和 §2。
- **如果用户内联提供内容**（XMind 内容、AI 需求、原始 PRD 文本）：将其视为内容输入。仍然检查并生成 init 模板——内联内容不替代背景上下文收集。
- **架构图（§5）必须在知识库查询之前处理**：读取/显示图，然后提取所有可见的组件/模块名称，在步骤 0.6 中将其添加为额外搜索词。图的内容在识别现有模块方面优先于 AI 推断。
- **§10 结构化补充信息处理**（在 §1 和 §2 确认填写后执行）：
  1. 检查 §10 是否填写"不涉及"→ 跳过
  2. 读取 §10.1–§10.4 四个子节的内容：
     - §10.1 数据模型字段需求 → 提取实体名、字段名称、类型约束、必填/唯一规则
     - §10.2 API 接口推导 → 提取页面/操作对应的 CRUD 接口清单和请求体结构提示
     - §10.3 枚举值与状态机 → 提取枚举取值列表、状态转换规则
     - §10.4 业务规则 → 提取校验规则、计算逻辑、权限控制规则
  3. 将上述提取结果存为**结构化补充数据**，在步骤 1 生成 REQ 描述和验收条件时引用

**⛔ 在 init 模板填写完成且用户确认之前不要进入 0.2。**

#### 0.2 ⛔ 硬关卡 —— Init 完整性校验

评估每个章节的**信息质量**（而不仅仅是非空）。提交一份完整性报告。**在所有阻塞项解决之前不要继续。**

| 章节 | 是否阻塞？ | 充分条件 | 不充分示例 |
|---|---|---|---|
| §1 业务背景 | **是** | 清晰说明：谁是用户、解决什么问题、为什么是现在 | "做这个功能" / 占位文本 |
| §2 关键业务场景 | **是** | 至少 1 个具体场景含：用户角色 + 触发条件 + 预期结果 | "用户使用该功能" / 泛泛描述 |
| §4 关联服务和系统 | 否 | 有具体的服务/模块名称 | 留空可接受 |
| §6 领域术语和业务规则 | 否 | 每个术语有定义 | 留空可接受 |
| §8 版本需求范围界定 | 否 | 有明确的"本版本包含"和"本版本不包含"列表 | 留空可接受 |
| 用户需求文档（如提供） | **是** | 文档可读、有章节结构、可解析 | 文件未找到 / 不可读 |

**报告格式：**
```
📋 Init 完整性校验结果
✅ §1 业务背景：[一句话概括]
✅ §2 关键场景：识别到 N 个场景
⚠️ §6 领域术语：未填写，后续将使用通用术语（非阻塞）
🚫 §1 业务背景：信息过于简单——请补充说明
```

如果任何阻塞章节不充分：告知用户具体缺少什么，等待更新，重新验证。非阻塞差距：注明并继续。

**⛔ 在所有阻塞项通过之前不要进入 0.3。**

#### 0.3 ⛔ 硬关卡 —— 知识归档

扫描 init 模板中跨以下类别的可复用领域知识候选项：

- **§1 业务模式**：可复用的业务模式（工作流、审批链、资源分配模型）——候选项类型：业务模式胶囊。
- **§6 领域术语**：具有非显而易见语义的领域特定术语——候选项类型：领域术语胶囊。
- **§2/§6 业务规则**：领域规则（状态机约束、计算公式、法规要求）——候选项类型：领域规则胶囊。
- **§4/§5 架构约束**：架构约束（技术锁定、部署限制、集成边界）——候选项类型：架构约束胶囊。

**扫描后，你必须向用户提交发现——即使什么都没找到：**

```
📦 知识归档候选项（Step 0.3）

从需求背景模板中扫描到以下可复用领域知识：
1. [业务模式] xxx 审批流程（多个子系统复用）
2. [业务规则] 状态机约束：xxx 不可逆
3. [领域术语] "xxx" 在本业务中特指 xxx

请选择要归档的序号（如 1 2 3），或输入"跳过"不归档。
```

若未扫描到任何候选项，仍需告知用户：

```
📦 知识归档候选项（Step 0.3）

未发现值得归档的可复用领域知识（业务模式、术语、规则、架构约束均为空或项目特异性内容）。

如有需要补充归档的内容，请现在告知；否则输入"继续"进入系统需求分析。
```

用户确认后，对选中条目调用 `daedalus-knowledge` 执行归档。

归档后："背景情报已校验完整，可复用知识已归档至团队知识库。开始系统需求分析。"

**⛔ 在归档确认完成之前不要进入 0.4。**

#### 0.4 加载输入需求

- **XMind 文件**（`.xmind` 或含 `content.json` 的文件夹）：读取 `content.json`/`content.xml`，提取需求树，将节点映射到 REQ 项。
- **AI 需求（Epic/Feature/Story）**：检查 `docs/agent-rules/1-ai-requirements/output/` 中最新的 `YYYY-MM-DD-{project}-requirements.md`。
- **结构化用户需求文档**（编号章节 + 至少包含以下两项：功能/非功能分类、验收标准、业务场景）：直接阅读，映射到 REQ 项。
- **原始 PRD**（无章节结构）：应先通过 `requirement-analysis` 路由。告知用户并在继续前请求确认。
- **仅 init 模板**（无单独的需求文档）：将 §2 和 §6 作为主要来源；§3/§4 作为补充上下文。

#### 0.5 历史版本检查

首先检查 init 模板的 §3。如果无历史文档，询问用户。如果之前版本存在，加载它进行变更差异分析。参见[历史版本处理](#历史版本处理)了解变更分类和呈现格式。

#### 0.6 查询产品知识库（深入跟进）

基于步骤 0.1 的预查询，使用来自用户已验证的 init 模板（§4、§5）中**确认的**模块/服务名称重新查询。深度定位 API 规格说明、架构文档和技术栈——这为 REQ 增量分析和 DFX 目标提供信息。用任何新发现更新 §9 和 §4。

**应用三模式检测**（参见[知识库发现](#知识库发现)）：
- **config.kb.skill / kb-query 可用** → 使用 §4 和 §5 中确认的模块/服务名称调用知识库 skill
- **config.kb.localPath 存在** → 对模块名称和 API 关键词 Grep 本地知识库文件
- **代码优先** → 对接口定义、API handler、数据模型 Grep 源文件

#### 0.7 读取模板

从 `config.templates["system-requirement"]`（默认：`templates/system-requirement-template.md`）加载模板文件并内化其完整 9 章结构。

#### 0.8 搜索 DFX 基线

DFX 基线按**优先级顺序**获取：

**优先级 1 —— TR1 用户需求文档（最高优先级，用户已验证）：** 如果步骤 0.4 中加载了 TR1 文档（来自 `docs/agent-rules/1-ai-requirements/output/`），提取其 §2.2 DFX 需求基线章节。这些值是用户确认的，优先于所有其他来源。

**优先级 2 —— 知识库查询（补充缺口）：** 调用 `daedalus-knowledge` 查询平台 DFX 约束（从 `cospowers.config.json` 读取 `SPEC_DEVELOPER_SERVER_URL`；如果未配置，跳过知识库查找并回退到本地 `rules/dfx/` 文件）。**应用三模式检测**获取额外上下文（参见[知识库发现](#知识库发现)）：
- **kb-query/local-kb** → 查询下列 DFX 约束关键词
- **代码优先** → 在代码库中搜索性能配置、SLA 注释、监控设置

| 查询关键词 | 目标章节 | 提取内容 |
|---|---|---|
| `性能指标` / `响应时间基线` | §4.1 性能要求 | 具体的 TPS/延迟目标 |
| `可用性要求` / `故障恢复` | §4.2 可靠性要求 | SLA 目标、RTO/RPO 基线 |
| `业务监控` / `监控指标规范` | §4.4 可运维性 | 业务指标注册模式 |
| `安全基线` / `鉴权规范` | §4.3 安全性要求 | 认证模式、数据分类规则 |
| `可测试性规范` / `测试工具接入` | §4.7 可测试性 | 标准测试工具和 CI 集成 |
| `可扩展性要求` / `水平扩展` | §4.5 可扩展性要求 | 横向/纵向扩展策略 |
| `兼容性要求` / `版本兼容` | §4.6 兼容性要求 | 浏览器/OS/数据库/向后兼容基线 |
| `开放性要求` / `标准化接口` | §4.8 开放性要求 | 开放 API 定义、认证、限流 |
| `日志规范` / `可观测性` | §4.9 日志与可观测性要求 | 日志级别、保留期、trace ID、监控指标 |

将结果预填到 `init-sysreq-template.md` §9。如果某查询无结果：标记 `[待确认：知识库无基线，请人工填写]`——不要替代为通用文本。

### 步骤 1：范围分析与用户确认

#### 1.1 提交需求项计划

展示从 Story 映射的 REQ-XXX 列表，如果存在历史版本则附变更标签（`[新增]` / `[变更]` / `[延续]` / `[废弃]`）。获取用户确认。

#### 1.2 推断缺失需求

基于已确认的 REQ 列表，推理通常需要但尚未说明的内容。一次问用户一个主题组。参见[需求完整性推断](#需求完整性推断)了解模式和规则。

#### 1.3 处理调整请求

迭代 REQ 项直到用户满意。

#### 1.4 确认生成范围

请用户选择三种生成模式之一：

- **A) 完整分析** —— 撰写全部 9 个章节文件，每个 REQ 项完整内容（全部优先级、完整验收标准、完整 DFX）。默认选择。
- **B) 核心优先（仅 P0）** —— 撰写全部 9 个章节文件，但仅完整开发 P0 REQ 项；P1/P2 项在 REQ 表中作为存根出现（描述和验收标准字段为 `[待完善]`），待后续填写。
- **C) 示例模式** —— 撰写 2–3 个 REQ 项作为完整工作示例（含完整验收标准和 DFX）；其余 REQ 项以存根列出。适用于早期对齐评审。

**⛔ 在用户确认 REQ 计划和生成范围之前不要进入步骤 2。**

### 步骤 2：生成系统需求文档

#### 2.1 创建输出目录和 index.md

**输出目录：** `docs/agent-rules/2-system-requirements/output/YYYY-MM-DD-<project>/`

**文件结构：**
- `index.md` —— 元数据表 + 章节目录及单行摘要（先写，摘要最后补充）
- `ch01-文档说明.md` —— §1 文档说明

#### 3.1 完整性验证

#### 3.2 跨章节一致性检查

1. **术语一致性** —— 从 §1.2 中选取关键术语，搜索所有章节。标记术语漂移（同一术语在不同含义下使用）。

4. **场景覆盖** —— 第 3 章中的每个 REQ 项必须链接到第 2 章中至少一个场景。
5. **术语重定义** —— 如果本版本更改了术语的含义，在 §1.2 中显式重新定义："本版本重新定义：原含义为X，现含义为Y"。

**每个 REQ 的 SMART 检查：** 具体（清晰描述）、可度量（具体验收标准）、可实现（可行、合理工作量）、相关（与业务目标对齐）、有时限（优先级已分配）。

**可测试性：** 验收标准使用 GWT 格式；每个 REQ 覆盖全部 5 种场景类型；性能指标可量化；第 5 章测试点覆盖所有 REQ 项。

## REQ 编写规则

**规则 1：REQ 描述不得包含实现决策。**

| 属于 REQ | 属于 overall-design-spec |
|---|---|
| "系统必须支持对用户操作的完整审计" | "使用 Elasticsearch 存储审计日志" |
| "接口响应时间 P95 < 200ms" | "通过 Redis 缓存降低响应时间" |
| "支持1000并发用户" | "采用消息队列异步处理" |

**规则 2：将未确认的需求标记为 `[待确认]`。**

如果需求未经产品负责人或客户明确确认，标记 `[待确认: 需与XX确认]`。未确认的需求不得视为 P0。

**规则 3：DFX 数字必须有真实来源——永远不要凭空编造。**

优先级顺序：(1) 团队知识库中的平台基线，(2) 历史版本实际值，(3) 询问用户。模板示例数字（例如 ">= 1000 并发用户"、"< 200ms P95"）是格式占位符，不是默认值。如果没有可用来源，标记 `[待确认]`——不要填入数字。

## 5 种场景类型

每个 REQ 项必须包含覆盖全部 5 种类型的验收标准：

1. **正常流程**（2-4 个场景）——具体的数据成功路径
2. **异常流程**（2-3 个场景）——错误输入、系统故障、网络问题
3. **安全性**（2 个场景）——权限验证、数据加密、访问控制
4. **边界**（2 个场景）——极限值、资源上限、并发限制
5. **可测试性**（1-2 个场景）——测试数据构造和验证方法

每个场景遵循 Given-When-Then 格式：
```markdown
##### [正常流程场景 1：成功创建模板]
**Given**：管理员已登录管理控制台，"服务模板"页面已加载，模板名称 "Web-App-Standard" 尚未使用
**When**：管理员点击"新建模板"，填写名称 "Web-App-Standard"，选择基础镜像 "Ubuntu 22.04"，设置资源限制 CPU=2 核 Memory=4GB，点击"保存"
**Then**：系统显示"模板创建成功"，模板列表显示新条目 "Web-App-Standard"，状态为"可用"，创建时间显示当前时间
```

## DFX 非功能性需求

全部 9 个核心维度必须处理：

| 章节 | 维度 | 关键指标 |
|---|---|---|
| **4.1** | 性能要求 | 并发量、响应时间、吞吐量、CPU/内存使用率；资源需求 |
| **4.2** | 可靠性要求 | 可用性（如 99.9%）、RTO、RPO、MTBF、MTTR；故障检测/恢复/备份 |
| **4.3** | 安全性要求 | 认证/授权、数据加密/脱敏/备份、安全防护、审计日志 |
| **4.4** | 可运维性要求 | 部署方式、监控/告警、日志管理、配置管理、升级/回滚 |
| **4.5** | 可扩展性要求 | 水平/垂直扩展能力 |
| **4.6** | 兼容性要求 | 浏览器、OS、数据库、向后兼容性 |
| **4.7** | 可测试性要求 | 接口可测试性、状态可观测性、数据可构造性、故障可注入性、环境隔离性、日志可追溯性 |
| **4.8** | 开放性要求 | 开放 API 定义、认证方式、限流策略、版本兼容承诺 |
| **4.9** | 日志与可观测性要求 | 操作/错误日志范围、日志级别、保留期限、trace ID 关联、监控指标 |
