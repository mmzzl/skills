# 模块设计规格说明

**Skill 标识**: `module-design-spec`

其他 skill 通过 `module-design-spec` 引用本 skill。

## 概述

将总体设计和 API 契约转化为模块级概要设计文档，严格遵循 `config.templates["module-design"]`（默认：`templates/module-design-template.md`）的模板。**每次调用处理一个模块**——多个模块负责人独立调用本 skill。

**输入：** 总体设计文档 + OpenAPI 规格说明（由 `overall-design-spec` 产出）
**输出：** 模块方案选型 + 一份模块概要设计文档，作为共享项目输出目录下的章节文件目录

**核心原则：** 先完成模块方案选型，再撰写模块概要设计正文：对手分析和方案选型结论必须先锁定。模块设计必须先承接总体设计任务，再展开模块内部实现；无空章节，必须有 Mermaid 图，数据库表使用 SQL DDL，DFX 覆盖为强制性要求且必须集成总设第 6 章，API 引用指向 OpenAPI 规格说明（不重复），产出可评审、可交给 `writing-plans` 拆解实现。

## 扩展点

开始之前，从插件根目录（此 skill 基目录向上 2 级——即 skill 加载时显示的"此 skill 的基目录"）读取 `cospowers.config.json`。无需回退——配置文件始终包含有效默认值。

| 配置字段 | 用途 |
|---|---|
| `config.templates["module-design"]` | 模块概要设计文档的模板文件路径 |
| `config.rules["design-review"]` | 设计评审规则目录 |
| `config.rules["dfx"]` | DFX 规则目录 |
| `config.evaluators["module-design"]` | 每个模块调度的评估器 skill 名称（步骤 8）；`false` = 跳过 |
| `config.evaluators["doc-quality"]` | 仅最后一个模块调度的评估器 skill 名称；`false` = 跳过 |

## 何时使用

- 总体设计 + OpenAPI 规格说明已存在，且你需要撰写自己负责模块的概要设计
- 各模块负责人独立调用本 skill 处理其被分配的模块
- 在端到端团队项目工作流中，紧随 `overall-design-spec` 之后
- 用户明确请求创建模块概要设计文档

**以下情况不要使用：**
- 总体设计和 OpenAPI 规格说明尚未定义（请先使用 `overall-design-spec`）
- 小微项目（直接使用 brainstorming → writing-plans）

## 检查清单

你必须为以下每一项创建任务，并按顺序完成：

1. **加载总体设计和 OpenAPI 规格说明** —— 扫描 `docs/agent-rules/3-overall-design/output/` 中前缀为 `YYYY-MM-DD-` 的子目录；取最新的（按降序排列）。

   **1a. 加载总体设计：** 阅读 `index.md` 了解产品、迭代和章节目录。提取：
   - 架构总览 —— 模块在整体系统中的位置和关系（总体设计 §2.2 系统架构图（4+1视图））
   - 模块静态结构 —— 模块层级、职责和依赖关系（总体设计 §2.3 模块划分、§2.4 模块依赖关系、§5.2 模块静态结构）
   - 模块设计要求 —— 哪些模块需要概要设计、设计目标和约束（总体设计 §5.4 模块设计要求）
   - 系统数据架构 —— 与目标模块相关的全局实体和跨模块数据结构（总体设计 §5.3 系统数据架构）
   - 系统级 DFX 方案 —— 读取总体设计第 6 章/`ch06-系统DFX特性设计.md`，提取与目标模块相关的安全、可靠、性能、可运维、可测试、可调试等系统级 DFX 架构决策、约束、指标和责任分配；后续 Ch.5 必须集成这些总设 DFX，不得重新独立发明一套 DFX 方案


   **1b. 加载 OpenAPI 规格说明：** 从同一目录读取 `<project>-openapi.yaml`。提取：
   - 按模块标签分组的所有端点定义（方法、路径、请求/响应 schema）
   - 错误码定义
   - API schema 定义中的类型约束

   **⛔ 硬关卡 —— OpenAPI 必须在继续前有效。**

   | 检查项 | 验证方式 | 是否阻塞？ |
   |-------|---------------|-----------|
   | 文件存在 | `ls` 路径 `docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/<project>-openapi.yaml` | **是** —— 缺失则停止 |
   | YAML 合法 | 解析；确认加载无错误 | **是** —— 非法则停止 |
   | 有端点 | 至少有一个 `paths:` 条目含方法 | **是** —— 为空则停止 |

   **如果任意检查失败：不要继续。** 告知用户：

   > "OpenAPI 规范文件缺失或无效：`docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/<project>-openapi.yaml`
   > [具体问题]
   > 请回到 `overall-design-spec` 修正 OpenAPI 后再调用 `module-design-spec`。"

   **1c. 选择目标模块：** 向用户提交 §2.3 中的模块列表：

   向用户提交模块列表，询问他们负责哪个：

   > "总体设计包含以下模块（来自 §2.3）：
   > - <module-1>
   > - <module-2>
   > - ...
   >
   > 你负责哪个模块的概要设计？"

   **本次调用范围限定为一个模块**。如果用户负责多个模块，他们分别调用本 skill 处理每个模块。

   **1d. 建立总设承接矩阵（后续写作的硬输入）：** 选择目标模块后，对该模块创建并持续维护一张承接矩阵，写入 `index.md` §1.5/§1.6 或作为 Ch.2 前置表格引用。矩阵至少包含：

   | 总设来源 | 总设要求/决策 | 本模块责任 | 落地章节 | 状态 |
   |---|---|---|---|---|
   | §2 架构/模块视图 | 模块职责、边界、依赖 | 本模块承担/不承担什么 | Ch.2 / Ch.4 | 已落实/不适用/待确认 |
   | §4 交互流程 | 跨模块调用、异常传播 | 本模块入口、出口、错误处理 | Ch.3 / Ch.4.3 / Ch.4.6 | 已落实/不适用/待确认 |
   | §5 数据架构 | 数据归属、共享实体、持久化边界 | 本模块拥有/读取/更新的数据 | Ch.4.5 / database-design.md | 已落实/不适用/待确认 |
   | §6 DFX | 系统级 DFX 决策和指标 | 本模块的具体机制与责任边界 | Ch.5 / Ch.6 | 已落实/不适用/待确认 |
   | §8 风险 | 与本模块相关的风险和缓解 | 本模块需规避或承接的风险 | Ch.8 | 已落实/不适用/待确认 |
   | §9 部署 | 部署单元、网络、配置、依赖服务 | 本模块部署/发布/回滚要求 | Ch.7 | 已落实/不适用/待确认 |

   **硬规则：** 后续章节不得绕开承接矩阵。总设中与本模块相关但未落地的条目必须标为 `[待确认]` 或说明“不适用及原因”；不得静默遗漏。

   **⛔ 硬关卡 —— 总设承接完整性。** 在进入步骤 2 之前，必须向用户展示承接矩阵摘要并确认：模块职责边界、关键接口、数据归属、DFX 责任、部署/风险约束是否承接正确。未确认前不要开始 init 模板收集和正文写作。

1e. **查询产品知识库** —— 在选择目标模块（步骤 1c）和建立总设承接矩阵（步骤 1d）之后、收集 init 模板（步骤 2）之前，使用所选模块名调用产品知识库，发现该模块的现有实现、API、数据模型、依赖、故障模式和技术债。

   **四级降级检测**（读取 `cospowers.config.json`，详见下方"知识库发现"章节）：
   - `config.kb.skill` 已设置 → 调用该 skill
   - 否则环境中 `kb-query` 可用 → 使用 `kb-query`
   - 否则 `config.kb.localPath` 存在 → Grep/Read 本地知识库文件
   - 否则 → 代码优先（Glob + Grep + Read）

   **查询目标（按模式映射）：**

   | 模式 | 操作 |
   |---|---|
   | **kb-skill / kb-query** | 按模块名查询：业务领域、上下游依赖、现有 API、现有数据模型、已知故障模式、已知 bug / 冻结代码 |
   | **local-kb** | Grep `<config.kb.localPath>` 下与模块名相关的所有 `*.md` 文件 |
   | **代码优先** | Glob 入口文件、handler/service/router/model；Grep 接口定义和数据模型 |

   **必须获取的信息：**
   - 模块业务领域与上下游依赖（与 1d 承接矩阵交叉验证）
   - 现有 API 清单（防止 Ch.3 重复定义）
   - 现有数据模型 / 表结构（防止 Ch.4.5 + database-design.md 与现有冲突）
   - 已知故障模式 / 排障知识（输入 Ch.4.6 异常处理 + Ch.5.2 FMEA）
   - 已知 bug / 技术债 / 冻结代码（输入 Ch.8 风险与遗留问题）

   **KB 结果 → 后续设计元素的映射：**

   | 知识库发现 | 输入到 |
   |---|---|
   | 现有 API | Ch.3 对外接口（避免重复定义） |
   | 现有数据模型 | Ch.4.5 数据结构 + database-design.md |
   | 故障模式 / 排障知识 | Ch.4.6 异常处理 + Ch.5.2 可靠性 FMEA |
   | 技术债 / 冻结代码 | Ch.8 风险与遗留问题 |
   | 业务领域 / 上下游 | 步骤 1d 承接矩阵（交叉验证）+ 步骤 2 init §5/§6 预填 |
   | 已有领域术语 | index.md §1.2 术语表 |

   **知识库指引方向，代码验证真相。** 在 Step 3a（读取代码）和 Step 4（撰写文档）时，必须用 Glob/Grep/Read 对照实际代码验证 KB 发现；当 KB 与代码不一致时以代码为准。

2. **通过 init 模板收集模块实现上下文** —— 针对选定的模块，收集负责人掌握的、总体设计中没有包含的实现约束：

   **Init 模板章节 → 模块概要设计映射：**

   | 章节 | 提取内容 | 输入到 |
   |---|---|---|
   | §0.1 对手分析 | 竞品/业界/内部历史方案 | 模块方案选型 §1 对手分析 |
   | §0.2 待选型的关键技术问题 | 待做的技术决策及 trade-off | 模块方案选型 §2 选型范围和对比维度 |
   | §0.3 选型评估的约束目标 | 评估标准和硬指标 | 模块方案选型 §2 评分准则和权重 |
   | §0.4 已有方案选型材料 | 已有分析、ADR、预研 | 模块方案选型依据；避免重复分析 |
   | §0.5 已确定的技术约束 | 总设/OpenAPI/代码中的硬约束 | 模块方案选型硬边界（不重新讨论） |
   | §1 模块基本信息 | 模块确认、仓库/服务归属、依赖关系 | `index.md` §1.4；Ch.2 职责和边界 |
   | §2 模块内部结构图 | 内部组件/子模块图 | Ch.4.1 子模块拆分；对照代码验证 |
   | §3 依赖文档 | 历史模块设计、ADR、API 文档 | 历史设计处理；参考资料 |
   | §4 技术债务和遗留问题 | 已知 bug、冻结代码、遗留约束 | Ch.2 边界、Ch.4 设计决策、Ch.8 风险与遗留问题 |
   | §5 业务规则和领域知识 | 代码中不可见的领域规则 | Ch.1 需求跟踪、Ch.4 流程、Ch.6 自测用例 |
   | §6 依赖的关键外部接口 | 已知 API 怪癖、限制、故障模式 | Ch.3.5 依赖接口；Ch.4.6 异常处理 |
   | §7 其他信息 | 任何额外上下文 | 应用于所有章节的背景知识 |
   | §8 涉及服务的本地代码路径 | 涉及服务的本地代码路径 | 代码验证；实现上下文 |

   - 检查 `docs/agent-rules/4-module-design/input/init-{module-name}-module.md` 是否已存在：
     - **如果存在且已填写**：阅读、验证，进入步骤 3
     - **如果存在但为空或部分填写**：告知用户还需要什么，等待完成
     - **如果不存在**：读取 `templates/init-module-design-template.md`，预填已知信息（模块名称、总体设计的设计要求、OpenAPI 中的 API 候选），写入上述路径，告知用户：

       > "我已为「{module-name}」生成背景情报简报，保存在 `docs/agent-rules/4-module-design/input/init-{module-name}-module.md`，请在 IDE 中打开填写。
       > **§0.1 对手分析为必填项**（无可比对象需明确写"无"并说明原因）；§1 基本信息为必填项；§2 内部架构图强烈建议提供；§4 技术债务务必如实填写。填写完成后告知我。"

   - **等待用户确认后再继续。**

   **⛔ 硬关卡 —— 步骤 2a：Init 完整性校验**

   读取填写好的 init 模板后，评估信息质量。注意：此关卡仅检查输入信息是否充分（能否开始设计？），而非设计质量。设计质量检查是步骤 8 的 module-design-evaluator 子代理。

   | 章节 | 是否阻塞？ | 充分条件 |
   |---|---|---|
   | §0.1 对手分析 | **是** | 列出竞品/业界/内部参考，或明确说明无可比对象及原因——留空不可接受 |
   | §1 模块基本信息 | **是** | 模块名称已确认，服务/仓库归属已说明，或明确未知并说明原因 |
   | §4 技术债务 | 否 | 列出已知问题或明确写"无已知债务"——留空可接受 |
   | §2 内部架构图 | 否 | 图已提供；若无，注明将从代码推断模块拆分 |
   | §5 业务规则 | 否 | 领域规则具体且可测试（不仅仅是"按业务需求"） |
   | §6 依赖的外部接口 | 否 | 每个列出的依赖都记录了已知怪癖/故障模式 |

   如果任何阻塞项失败：告知用户具体需要什么，等待更新，重新读取并重新验证。**在阻塞项通过之前不要开始撰写。**

   **⛔ 硬关卡 —— 步骤 2b：知识归档到 daedalus-knowledge**

   完整性校验通过后，扫描输入文档中跨四个类别的可复用知识候选项：

   - **技术债务**（§4）：具体的冻结代码区域、已知 bug 及变通方案、遗留约束（AI 必须避免触及）——候选项类型："禁区"胶囊。
   - **业务规则**（§5）：代码中不可见的领域规则（状态机约束、计算公式、法规要求）——候选项类型：领域规则胶囊。
   - **外部接口怪癖**（§6）：非显而易见的 API 行为、速率限制、依赖的故障模式——候选项类型：集成模式胶囊。
   - **内部架构决策**：§2 中揭示的非显而易见设计选择（例如，"此服务使用事件溯源进行审计，而非 CRUD"）——候选项类型：架构决策胶囊。

   **扫描后，你必须向用户提交发现——即使什么都没找到：**

   ```
   📦 知识归档候选项（Step 2b）

   扫描到以下内容值得归档：
   1. [技术债务] xxx 模块的 xxx 方法不可修改（原因：xxx）
   2. [业务规则] 状态机约束：xxx 状态不能直接跳转到 xxx
   3. [架构决策] 本服务使用事件溯源而非 CRUD（原因：xxx）

   请选择要归档的序号（如 1 2 3），或输入"跳过"不归档。
   ```

   若未扫描到任何候选项，仍需告知用户：

   ```
   📦 知识归档候选项（Step 2b）

   未发现值得归档的内容（无技术债务、业务规则、接口怪癖或非显而易见的架构决策）。

   如有需要补充归档的内容，请现在告知；否则输入"继续"进入设计阶段。
   ```

   用户确认后，对选中条目调用 `daedalus-knowledge` 执行归档。归档完成后进入步骤 3。

3. **完成模块方案选型** —— 在撰写模块概要设计正文前，基于总体设计、init 输入、历史材料、知识库和代码扫描，完成独立的模块方案选型。

   - 读取模板：`templates/design/module-design/模块方案选型.md`
   - 输出到：`docs/agent-rules/4-module-design/output/YYYY-MM-DD-<project>/<module>/module-option-selection/`

   **必填内容：**
   - §1 对手分析：保留模板结构，填写竞争对手、业界方案或内部历史方案；如果没有可比对象，明确写"无可参考对手/业界方案"并说明原因。
   - §2 方案选型：保留模板结构，对影响架构、算法、存储、通信、依赖处理、DFX 或实现风险的关键模块级决策完成方案选型。

   **此步骤需要交互。** 持续追问，直到对手分析和方案选型足以支撑模块设计。如果用户已在 init §3 或依赖文档中提供足够材料，应吸收这些材料，而不是要求重复填写。

   **⛔ 硬关卡 —— 步骤 3a：模块方案选型完成度**

   在模块方案选型足以锁定关键决策前，不要撰写 Ch.2-Ch.9。模块方案选型必须像总设 TRD 一样成为正文设计的决策来源，而不是可选附件。

   | 检查项 | 是否阻塞？ | 充分条件 |
   |---|---|---|
   | 对手分析 | 否 | §1 包含竞争对手/业界/内部参考，或明确说明没有可比对象及原因 |
   | 总设边界继承 | **是** | 明确列出哪些总设决策是硬边界，不重新讨论；若要偏离，必须有用户确认依据 |
   | 关键方案选型 | **是** | 每个影响架构、算法、存储、通信、依赖处理、DFX、可测试性或实现风险的模块级决策都有背景、备选方案、对比和结论；或文档明确写"无关键方案选型，本模块按总体设计既定方案实现" |
   | 决策落地映射 | **是** | 每个选型结论都标注将落到 Ch.2/Ch.3/Ch.4/Ch.5/Ch.6/Ch.7/Ch.8 的哪个章节 |

## 设计阶段 AI 角色

设计质量来源于在每个阶段有意识地切换正确的模式。

### 作者模式
按团队模板生成设计文档章节。产出文档内容时的默认模式。

### 挑战者模式
在整个写作过程中应用。对每个设计决策：
- 问："这个选择可能会出什么问题？"
- 问："我假设了什么尚未确认的东西？"
- 问："是否有我更简单但被过快否决的替代方案？"
- 在确定第一个想法之前，强制提出一个替代方案。

将未确认的假设内联标记为 `[待确认: reason]`。

### 评审员模式
它定义了子代理评审员（步骤 7）会检查什么。这些也应成为作者在写作过程中始终自查的标准。

**完整性检查（缺少什么）：**
- 系统需求中的每个 REQ-XXX 是否有对应的设计元素？
- 不同的团队能否在无需提问的情况下实现此设计？
- 是否有任何章节描述不足（"参考现有实现"但没有细节）？

**严谨性检查（有什么但不应被信任）：**
- **伪精确数字**：每个具体数字必须有可验证的来源。如果没有，替换为范围或标记 `[待验证: 无基准数据，需压测确认]`。
- **跨章节矛盾**：当发现一处不一致时，扫描所有相关章节——单个发现的冲突通常隐藏着集群。成组修复。
- **选型理由**：仅比较表不够。每个标准是否有可验证的依据？被拒绝的方案是否用具体的技术理由解释了？
- **客观的章节命名**：设计文档呈现解决方案——而不是自我推销。"独有的设计创新" → 重构为分析（"领域需求 vs 框架能力差距"）。
- **无依据的接口列表**：在实现之前通过分析得出的 API 列表是推演，而非设计承诺。标注为"初步规划（待实现验证）"，并单独记录架构原则。

**每个断言的核心问题："证据是什么？"——如果无证据，要么补充，要么标记为 `[待验证]`。**

### 设计大师模式

**何时调用：** 在模块方案选型和模块概要设计撰写期间，特别是方案选型、Ch.2 边界/选型、Ch.4 内部设计和 Ch.5 DFX，使用 `design-master-perspective` 作为额外的质量过滤器。设计大师在模块级别特别有价值——许多团队反模式（过度耦合、过度设计、边界违反）在此层级最为明显。

**如何调用：** 使用 `Skill` 工具调用 `design-master-perspective`。将当前的设计章节或决策提交给设计大师进行诊断分析。设计大师将返回：诊断 → 根因 → 建议 → 置信度。

**关键集成点：**

| 设计阶段 | 咨询内容 | 设计大师视角 |
|---|---|---|
| 步骤 3，方案选型 §2 | 方案选型 | 选型依据、隐藏风险、评估准则 |
| 步骤 4，Ch.2 | 模块边界和方案结论 | **边界纪律**："此模块是否有单一明确的职责？是否有违反边界的间接依赖？" |
| 步骤 4，Ch.4 | 子模块分解和内部流程 | **金字塔分解**："子模块粒度是否合适？每个子模块的范围是否 MECE？" |
| 步骤 4，Ch.4.6 | 异常处理 | **错误截断启发式**："每个错误是否在最早可能的位置被检测？错误传播是否有界？" |
| 步骤 4，Ch.5 | DFX 章节草稿 | **易变/不变分离**："模块中哪些部分是稳定机制，哪些是可变策略？" |
| 步骤 8 | 调度评估器之前 | **反模式扫描**：筛查常见模块级反模式 |

**挑战者模式防止设计期间产生糟糕的决策。评审员模式捕捉缺少的内容和不应被信任的内容。设计大师模式作为第三道过滤器，加入了团队积累的设计智慧。三者均为强制性的。**

## 设计判断 vs 假设

每个设计决策要么是**锁定判断**，要么是**假设**。

### 锁定判断
由以下之一支撑：
- 知识库文档（通过团队知识库验证）
- 系统需求文档中的明确说明
- 总体设计或 OpenAPI 规格说明
- `rules/` 文件中的团队标准
- 本次会话中的用户确认

将锁定判断写为陈述性语句："该模块使用 Redis 进行会话存储。"

### 假设（[待确认]）
在没有明确依据的情况下做出的设计决策。内联标记如下：

> 缓存将容纳最多 10000 条记录。**[待确认: 无知识库基线，无历史数据——最终确定 DFX 章节前请与用户确认]**

**规则：**
- 永远不要将假设呈现为判断
- 每个 `[待确认]` 条目必须在设计标记为已批准之前提交给用户
- DFX 数字几乎总是假设——如果知识库中没有值且无历史基线可用，请询问用户。**永远不要将模板示例值当作真实需求使用。**
- 关于现有系统行为的假设必须对照代码（不仅是知识库）进行验证

## 增量对齐

设计文档分层撰写，设有用户检查点——而不是一次性完整草稿倾泻。每个检查点是一个自然的决策边界：如果这里有错误，修复起来成本低。如果跳过检查点在错误的基础上再写 5 章，返工成本就高了。

**不要撰写整份文档后再在最后请求批准。** 在有意义的块之后询问。

### 模块级设计对齐点

| 检查点 | 覆盖章节 | 确认内容 |
|-----------|-----------------|-----------------|
| **总设承接检查点** | 步骤 1d + index.md §1.5/§1.6 | 总设职责边界、关键接口、数据归属、DFX 责任、部署/风险约束是否承接正确？ |
| **接口检查点** | 第 2-3 章（职责边界 + 对外接口） | 职责边界正确？外部接口完整？依赖正确识别？OpenAPI 契约是否只引用不重复定义？ |
| **内部设计检查点** | 第 4 章（内部设计） | 子模块分解合理？关键流程覆盖主要场景？数据结构合适？是否违背总设数据架构/交互流程？ |
| **DFX 落地检查点** | 第 5 章（DFX 特性设计） | 总设第 6 章的 DFX 输入是否逐项落到模块机制、接口、配置、日志、监控和自测锚点？ |
| **完整文档评审** | 全部章节 | 一致性检查前的最终评审 |

### 如何表述检查点

完成检查点的章节后，写道：

> "第 2-3 章已写——涵盖 [关键决策的概要]。在继续之前需要确认几件事：
> 1. [具体问题 1]
> 2. [具体问题 2]
> 这些看起来对吗，还是我应该调整什么？"

保持检查点消息简短。不要总结每一句——只总结用户需要验证的决策。

## 历史设计处理

如果存在之前的模块设计文档，将每个主要章节和接口分类并标记变更标签：

| 标签 | 含义 |
|-----|---------|
| `[延续]` | 与之前设计相同，未改变 |
| `[新增]` | 本轮新增，无之前的对应项 |
| `[变更-来自§X.X]` | 相对于之前设计第 X.X 节已修改——描述变更内容 |
| `[废弃]` | 之前存在，本轮移除——解释原因 |

**修订记录要求（Ch.8 变更控制）：**
每个修订条目必须说明变更内容及其原因，而不仅仅是列出版本号：

```
| V2.0 | 张三 | 2026-04-09 | 新增用户权限模块（来自REQ-012），废弃旧ACL实现（安全合规要求）|
```

"更新设计"或"按需求修改"不是可接受的修订描述。

## 3 条设计红线

这些是**强制性的**，必须在标记模块设计完成前验证：

### 红线 1：章节完整性

**设计文档中任何章节不得为空。** 每个章节必须包含实质性内容。如果章节不适用，必须明确说明"不涉及"并附上理由。此规则在模板顶部声明。

### 红线 2：可靠性章节

设计**必须**包含可靠性覆盖面：
- 功能承载组件的容错分析
- 资源使用分析（无不受控制的资源消耗）
- 业务流程可靠性（每条业务流程可靠）
- 基于容错思维，而非代码缺陷分析

### 红线 3：安全章节

设计**必须**包含安全覆盖面：
- 安全基线机制集成
- 威胁建模分析（如果安全经理要求）
- 基于已识别风险的安全设计
- 使用前组件的版本合规性和漏洞状态

## 强制模板元素

生成模块概要设计文档时，模板中的以下元素是**强制性的**，不得跳过或简化：

### Mermaid 图（模板中以星号标记为必填）

所有在模板中标记为 ⭐必填 的章节都需要 Mermaid 或 PlantUML 图。纯 ASCII 图明确禁止。每个图后必须跟随结构化文本描述表。

**模块级设计必填图：**
- Ch.4.1 子模块依赖图
- Ch.4.3 流程图，使用包含正常/异常分支（alt/else）的 sequenceDiagram
- Ch.7.1 部署架构图（部署在范围内时）

### 数据库表的 SQL DDL

Ch.4.5.1 要求在涉及持久化存储时使用 SQL DDL（非 Markdown 表格）定义数据库表。每个 DDL 必须包括：
- 每列 COMMENT 解释其含义
- 枚举字段列出所有合法值
- 索引注释解释它们服务于哪些查询
- 表级注释及关联的代码仓库注解

### AI 可读性注解

模板包含 `[AI可读性要求]` 和 `[AI读取引导]` 注解，定义了结构化输出要求。这些必须精确遵循：
- 每个流程必须同时包含图（给人看）和步骤描述表（给 AI 看）
- 接口引用必须使用标准格式：`接口名 (HTTP方法 路径)，详见《API Schema文档》§X.X.X`
- 配置文件必须指定：存储路径、格式、字段类型、默认值、取值范围以及是否需要重启

### 文档元数据表

模板以结构化元数据表开头，必须完整填写，包括：文档类型、模块/子模块名称、所属系统名称、迭代版本、输入/输出文档、代码仓库、关键词。

## DFX 维度

每个模块概要设计文档**必须**在 Ch.5 中处理 9 个 DFX 维度。模块级 DFX 必须集成总体设计第 6 章：总体设计给出系统级架构方案和跨模块约束，模块设计给出本模块的具体实现、配置、接口、日志、监控、自测用例与责任边界。仅在提供具体理由时使用"不涉及"。

| 维度 | 模板章节 | 关键内容 |
|---|---|---|
| 安全 | 5.1 | 基线覆盖、威胁建模、安全设计、组件合规 |
| 可靠 | 5.2 | 载体/进程/数据/业务可靠性和 FMEA |
| 可测试 | 5.3 | 依赖注入、测试接口 |
| 可调试 | 5.4 | 调试信号、诊断能力、可追踪性 |
| 可运维 | 5.5 | 日志、监控指标、部署运维 |
| 可扩展 | 5.6 | 插件机制、配置驱动扩展 |
| 可复用 | 5.7 | 可复用模块、公共代码、设计模式 |
| 隐私 | 5.8 | 隐私数据处理 |
| 跨平台 | 5.9 | 平台差异和兼容性 |

**重要模板规则（撰写模块级设计时适用）：**
- 流程设计章节不得包含代码示例；代码示例仅允许在核心算法章节中使用
- 步骤表中的外部调用必须标注 `[外部]`，DEP-XXX 引用编号与 Ch.3.5 匹配
- 当模块具有大量 CRUD 接口时，应按照模板的拆分规则拆分为子模块子文档
