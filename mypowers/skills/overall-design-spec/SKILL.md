# 总体设计规格说明

**Skill 标识**: `overall-design-spec`

其他 skill 通过 `overall-design-spec` 引用本 skill。

## 概述

将系统需求转化为严格遵循团队标准模板的正式设计文档。按顺序完成三类文档产出/阶段：

1. **技术需求设计**（TRD，作为独立前置输入产出）——澄清问题定义、场景分析、长期演进、竞品/对标分析、方案选择及决策依据
2. **总体设计**（模板：`config.templates["overall-design"]`，默认 `templates/overall-design-template.md`）——系统范围架构、模块/子系统拆分、跨模块交互及系统级 DFX
3. **API/接口规格说明**（模板：`config.templates["openapi"]`，默认 `templates/openapi-template.yaml`）——基于总体设计定义模块/子系统之间的契约

**产出顺序至关重要：** init 收集背景约束与 TRD 输入 → TRD 锁定问题定义和方案选择结论 → 总体设计定义有哪些模块/子系统以及它们如何交互 → API 规格说明锁定契约 → 模块概要设计（`module-design` skill）在这些契约下展开内部实现设计。

**核心原则：** 无空章节，所有标记位置必须有 Mermaid 图，DFX 覆盖为强制性要求，设计可追溯到需求，产出可评审。

## 扩展点

开始之前，从插件根目录（此 skill 基目录向上 2 级——即 skill 加载时显示的"此 skill 的基目录"）读取 `cospowers.config.json`。无需回退——配置文件始终包含有效默认值。

| 配置字段 | 用途 |
|---|---|
| `config.env.SPEC_DEVELOPER_SERVER_URL` | 远程知识库服务器 URL，用于知识发现 |
| `config.kb.skill` | 要调用的知识库 skill（null = 使用 kb-query 自动检测） |
| `config.kb.localPath` | 本地知识库目录，用于基于文件的查找 |
| `config.templates["overall-design"]` | 总体设计文档的模板文件路径 |
| `config.templates["openapi"]` | OpenAPI 规格说明的模板文件路径 |
| `config.rules["design-review"]` | 设计评审规则目录 |
| `config.rules["dfx"]` | DFX 规则目录 |
| `config.evaluators["overall-design"]` | 在步骤 12 调度的评估器 skill 名称；`false` = 跳过关卡 |

## 何时使用

- 系统需求已存在，且需要技术需求设计 + 总体设计文档
- 需要正式设计评审合规的项目
- 用户明确请求创建总体设计、特性级总体设计、技术需求设计或 OpenAPI 规格说明
- 端到端团队项目工作流中，在 system-requirement-analysis 之后

**以下情况不要使用：**
- 系统需求尚未定义（请先使用 system-requirement-analysis）
- 只需要模块概要设计（请改用 `module-design`）
- 小微项目（直接使用 brainstorming → writing-plans）

## 检查清单

你必须为以下每一项创建任务，并按顺序完成：

1. **加载系统需求** —— 阅读需求文档，理解所有 Story 和 REQ 条目。扫描 `docs/agent-rules/2-system-requirements/output/` 中前缀为 `YYYY-MM-DD-` 的子目录；取最新的（按降序排列）。读取该目录中的 `index.md` 以了解章节结构和概要。然后完整阅读 `ch03-功能性需求.md` 和 `ch07-设计任务书.md`；根据需要阅读其他章节。如果未找到子目录，请用户提供文档路径。
2. **检查历史设计文档** —— 如果之前的设计文档存在，记下曾存在哪些子系统/API/流程（参见下方"历史设计处理"章节）
3. **查询产品知识库** —— 先使用 `cospowers.config.json` 检测模式：config.kb.skill 设置 → 使用该 skill；否则 kb-query 可用 → 使用 kb-query；否则 config.kb.localPath 存在 → Grep 本地文件；否则 → 代码优先（Glob + Grep + Read）。参见下方"知识库发现"章节。
4. **搜索知识库中的 DFX / 平台约束** —— 调用 `daedalus-knowledge` 查询与本设计相关的平台 DFX 约束、设计指南和技术标准。该 skill 从 `cospowers.config.json` 读取 `SPEC_DEVELOPER_SERVER_URL`；如果未配置，则优雅降级（跳过知识库查找，不阻塞）。搜索关键词：系统名称、技术栈关键词、相关 DFX 维度（安全、可靠性、性能等）。
5. **加载模板** —— 在写作前加载并内化 TRD 模板结构和总体设计模板结构
6. **通过 init 模板收集总体设计约束** —— 在提出架构方案或编写 TRD 之前，收集只有人能提供的现有约束、历史决策和架构上下文：

   **Init 模板章节 → 设计步骤映射（处理填写好的模板时始终应用）：**

   | 章节 | 提取内容 | 输入到 |
   |---|---|---|
   | §0.1 本次设计类型 | 架构型 TRD 或特性型 TRD | 步骤 7：选择正确的 TRD 分支和文档范围 |
   | §0.2 技术需求设计范围 | 范围、关键问题、子系统、决策边界 | 步骤 7：TRD 访谈重点和章节覆盖 |
   | §0.3 已有技术需求材料 | 既有 TRD / 预研 / 评审输出 | 步骤 7：复用已有结论，避免重复分析 |
   | §0.4 已确定的技术选型 | 明确技术选项 | 步骤 7：**硬约束** —— 不得重新提案已确定的技术选项 |
   | §1 现有架构与系统资料 | 场景、逻辑、开发、运行、物理视角相关的现有图表、文档或文字说明 | 步骤 3：用资料中的组件名搜索；步骤 8：作为第 2 章 4+1 视图输入 |
   | §2 依赖文档 | 之前设计 / ADR / API / 评审文档路径 | 步骤 2：加载历史设计；保留既有约束和决策 |
   | §3 不能改动的服务 | 服务名 / 表名 | 步骤 3：增加知识库搜索词；步骤 7：硬约束 |
   | §4 外部集成 | 外部系统名称与数据流 | 步骤 7-8：总体集成设计、部署设计 |
   | §5 性能规模约束 | 具体数值（QPS、可用性等） | 步骤 7-8：DFX 目标 —— 使用用户提供数值，不使用模板示例值 |
   | §6 硬件方案输入 | 硬件 / 芯片 / 设备输入 | 步骤 8：总体设计第 3 章硬件方案 |
   | §7 其他约束 | 自由文本 | 步骤 7-8：如不可验证则标记为 `[待确认]` |
   | §8 本地代码路径 | 各服务的本地根目录路径 | 步骤 7（提出方案前）：对每个路径执行 Glob/Read，扫描入口文件、核心模型文件和路由文件，在提案前建立对现有实现的准确理解 |

   - 从 skill 的基目录读取 `templates/init-overall-design-template.md`
   - 根据系统需求和知识库结果预填已知章节：
     - §0.1 / §0.2：从系统需求中推导设计类型和大致范围
     - §0.3：补充已知预研 / 评审 / ADR 参考
     - §0.4：补充系统需求或知识库中已可见的技术约束
     - §3：补充步骤 3 中识别出的现有服务
   - 将预填内容写入 `docs/agent-rules/3-overall-design/input/init-overall-design-template.md`（如果目录不存在则创建）
   - 告知用户：
     > "我已将总体设计前置信息写入 `docs/agent-rules/3-overall-design/input/init-overall-design-template.md`，请在 IDE 中打开填写。**§1 现有架构与系统资料建议提供**（用于识别现有模块、流程、部署和接口，避免重复设计）。**§8 涉及服务的本地代码路径建议填写**（AI 将先读取代码再完成技术需求设计和总体设计，避免与现有实现冲突）。填写完成后告知我。"
   - **等待用户确认。在文件返回之前不要提出任何架构方案，也不要编写 TRD。**
   - 用户确认后：读取 `docs/agent-rules/3-overall-design/input/init-overall-design-template.md` 并应用上述映射表。现有架构与系统资料（§1）必须首先读取/显示 —— 提取所有可见的组件名称，在步骤 7 之前将其添加为搜索词。
   - 如果 §1 和 §2 均为空（完全没有架构上下文）：继续步骤 7，但将所有推断的子系统假设标记为 `[待确认]`。

   **⛔ 硬关卡 —— 步骤 6a：Init 完整性校验**

   读取填写好的 init 模板后，评估**信息质量**（不仅仅是是否有内容）。提交完整性报告，**在所有阻塞项解决之前不要进入步骤 7。**

   | 章节 | 是否阻塞？ | 充分条件 |
   |---|---|---|
   | §0.1 本次设计类型 | **是** | 明确分类为架构型 TRD 或特性型 TRD |
   | §1 现有架构与系统资料 或 §2 依赖文档 | **是（至少一项）** | 资料/文档可读且显示现有子系统边界或历史设计上下文；两项为空意味着 AI 将做出过多无法验证的假设 |
   | §0.4 已确定的技术选型 | 否 | 每个列出的选择有一行理由（避免就已有结论的决策反复讨论） |
   | §3 不能改动的服务 | 否 | 服务或表名具体明确（不仅仅是"有些服务不能动"） |
   | §5 性能规模约束 | 否 | 至少一个数字目标（QPS / 并发 / 延迟）—— 如未提供则标记为 `[待确认]` |
   | §8 本地代码路径 | 否（软阻塞） | 每个列出的服务路径在磁盘上确实存在（通过 Glob 验证）；如果路径未找到，在继续之前告知用户 |

   **完整性报告格式：**
   ```
   📋 总体设计 Init 完整性校验
   ✅ §0.1 设计类型：架构技术需求设计 / 特性技术需求设计
   ✅ §1 架构资料：已提供 [资料类型]
   ✅ §3 技术选型：已确定 N 项
   ⚠️ §6 性能约束：未填写，DFX 章节数值将标 [待确认]（非阻塞）
   🚫 §1 + §2 均为空：无法识别现有子系统边界，请至少提供其中一项
   ```

7. **完成技术需求设计（TRD）** —— 基于 init 输入、历史材料、知识库和代码扫描结果，撰写完整的 TRD，涵盖场景分析 → 方案选择 → 结论。TRD 是一个完整的交付物——方案选择过程和最终架构推荐都是文档的一部分，不另设独立步骤。

   - 阅读 `templates/design/overall-design/technical-requirements-design/1-技术需求设计.md`
   - 根据 init 的 `§0.1` 分流：
     - 架构型 TRD → 使用 `1-1-架构技术需求设计.md`
     - 特性型 TRD → 使用 `1-2-特性技术需求设计.md`
     - 如果两者都适用，先请用户选择一个主要 TRD 类型，再开始写作
   - 将 TRD 作为独立交付物写入 `docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/technical-requirements-design/`
   - 必需文件集：
     - `1-技术需求设计.md`
     - `1-1-架构技术需求设计.md`（当架构型 TRD 适用时）
     - `1-2-特性技术需求设计.md`（当特性型 TRD 适用时）
   - 此步骤是**交互式的**：持续追问，直到场景分析、长期演进、目标指标、方案选择依据和预研状态足以支撑后续设计。
   - 如果用户已经在 init `§0.3`（已有 TRD 材料）或 `§0.4`（已定技术选型）中提供了充分材料，应直接吸收，而不是强制重新录入。
   - 撰写 TRD 过程中，将**挑战者模式**（见"设计阶段 AI 角色"）和 **DFX 6 维度**（见"探索设计方案"）作为内部质量工具使用，不作为独立子步骤。引用知识库结论和 init 中全部硬约束（§0.4、§3）。未确认项统一标记为 `[待确认]`。

   **⛔ 硬关卡 —— 步骤 7a：TRD 完成门槛**

   在 TRD 足够回答下列问题之前，**不得进入总体设计撰写**：
   - 为什么需要这次设计工作
   - 哪些业务 / 架构目标最重要
   - 考虑过哪些候选方案
   - 为什么最终选择当前方案

   提交 TRD 完成摘要，并在继续之前获得用户批准。
8. **撰写总体设计文档** —— 遵循 `config.templates["overall-design"]` 对应模板（默认：`templates/overall-design-template.md`）（主入口 + 10 个子文档 / 子目录）。按增量对齐检查点写作：在架构（第 2+4 章）、模块设计说明（第 5 章）、DFX（第 6 章）、风险分析（第 8 章）和部署（第 9 章）后暂停，等待用户确认再继续。输出目录为 `docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/` 下的文件目录。

   **4+1 视图处理：**
   - 第 2 章是全文唯一的架构总图章节。4+1 视图按需选用，若部分视图重要程度不高可省略并说明原因，不适用的视图标注"不涉及"及理由（与模板 §2.2 规则一致）。
   - 后续章节不重复放置 4+1 图，直接引用第 2 章中的对应视图。
   - 建议图类型：逻辑视图 / 物理视图 / 开发视图使用 Mermaid `flowchart` 或 `graph`；过程视图 / 场景视图使用 Mermaid `sequenceDiagram`；禁止使用 ASCII 图。
   - 如果 init `§1` 某一视角输入缺失，在第 2 章对应视图中用 `[待确认]` 标记相关假设。

   **文件结构（总体设计包）：**
   - `index.md` —— 元数据 + §1 介绍（内联，不单独拆文件）+ 章节目录与单行摘要（先建目录，摘要后补）
   - `technical-requirements-design/` —— 复制步骤 7 产出的 TRD 文件，纳入最终交付包
   - `ch02-系统总体架构.md` —— §2 系统总体架构（4+1 视图、模块划分、依赖关系、技术选型）
   - `ch03-硬件方案.md` —— §3 硬件方案
   - `ch04-系统工作原理.md` —— §4 系统工作原理
   - `ch05-模块设计说明.md` —— §5 模块设计说明（含 §5.1 设计决策记录、§5.3 系统数据架构）
   - `ch06-系统DFX特性设计.md` —— §6 系统DFX特性设计
   - `ch07-对软件总体架构的影响.md` —— §7 对软件总体架构的影响
   - `ch08-方案风险分析.md` —— §8 方案风险分析
   - `ch09-部署视图.md` —— §9 部署视图
   - `ch10-总结.md` —— §10 总结
   - `ch11-变更控制与修订记录.md` —— §11 变更控制与修订记录
   - `<project>-openapi.yaml` —— OpenAPI 3.0 接口规范（由步骤 9 生成）

   **撰写顺序：**
   1. 创建 `index.md`，填入元数据表和章节列表（每个章节摘要先标记为 `[待写入]`）
   2. 在 §1.5 中概括已批准 TRD 的结论
   3. 逐章编写章节文件，遵循增量对齐检查点
   3. 全部章节完成后，回填 `index.md` 中的章节摘要（关键数量、图表数量）

   **`index.md` 必需内容：**
   - 文档元数据表（模板头部全部字段）
   - 高密度概要：采用了什么总体架构方案、拆成多少模块/子系统、关键 DFX 目标
   - 章节目录：`- [§N 标题](chNN-filename.md) — 单行描述`
   - 全局约束提醒："所有章节不得为空；不适用章节必须说明原因"

9. **撰写 API/接口规格说明** —— 输出为 **OpenAPI 3.0 YAML**，遵循 `config.templates["openapi"]`（默认：`templates/openapi-template.yaml`）。先提交端点列表并获得用户确认，再补完整 schema。将项目的类型系统约束应用到字段参数（通用模式见下方"OpenAPI 3.0 规格说明要求"）。保存到 `docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/<project>-openapi.yaml`（与总体设计同目录，不是并列目录）。

   **⛔ 硬关卡 —— 步骤 9a：OpenAPI 自检（进入下一步前必须通过）**

   撰写 OpenAPI 后，必须验证其完整性和正确性：

   | 检查项 | 验证方式 | 是否阻塞？ |
   |-------|---------------|-----------|
   | 文件存在 | `ls` 输出路径 | **是** —— 缺失则停止 |
   | YAML 合法 | 解析文件，确认加载无错误 | **是** —— 非法则停止 |
   | 端点覆盖 | 对照总体设计 §2.3.1 模块/子系统列表：每个拥有 API 的模块/子系统都至少有一个端点 | **是** —— 若某模块/子系统端点数为 0，且 §2.3.1 中又无 `[无对外API]` 标注，则停止 |
   | 必有字段 | 每个端点都包含 `operationId`、`summary`、`parameters`（如适用）、`responses`（至少 200 和错误码） | **是** —— 任一点不完整则停止 |
   | Schema 定义 | 所有 `$ref` 都能解析到 `components/schemas` 中已定义的 schema | **是** —— 有孤立引用则停止 |

   在进入步骤 10 之前先提交自检结果：
   ```
   🔍 OpenAPI 自检结果
   ✅ 文件存在: docs/agent-rules/3-overall-design/output/2026-05-06-xxx/xxx-openapi.yaml
   ✅ YAML 合法
   ✅ 端点覆盖: N 个子系统 / M 个端点（子系统 auth: 5, api: 3, ...）
   ✅ 必有字段完整
   ✅ Schema 引用全部可解析
   ```
   **如任一阻塞检查失败：必须修复 OpenAPI，重新执行自检，直到全部通过。**
10. **跨章节一致性检查** —— 验证 TRD 到总体设计的追溯关系、DFX 数值、需求可追溯性、术语一致性，以及总体设计文档内部的测试用例追溯关系（见下方"跨章节一致性检查"）。每个改变行为的 REQ 条目和 API 契约，都必须在第 6 章或相关章节中映射到至少一个可执行测试用例，覆盖正常、边界、异常或回归行为；缺口必须在评审前补齐。
11. **子代理评审** —— 使用 `skills/overall-design-spec/agents/design-document-reviewer-prompt.md` 调度一个子代理评审员，并在步骤 12 前修复发现的问题。
12. **质量评估（阶段关卡）** —— 使用 `config.evaluators["overall-design"]` 中的 skill 名称（默认：`overall-design-evaluator`），通过 `skills/<evaluator>/agents/evaluator-dispatch-prompt.md` 调度质量评估子代理。子代理在隔离上下文中运行，返回简洁等级 + 问题列表。必须修复所有 Critical/Error 问题。如果文档得分 < B（< 80），则向用户展示问题列表，并指出需要补充哪些 init 模板章节或 TRD 章节，然后从步骤 7 重新开始。未达到 B 之前不得继续。

    如果 `config.evaluators["overall-design"]` 为 `false`，跳过此关卡，直接进入步骤 13。
13. **用户评审设计** —— 请用户评审总体设计和 OpenAPI 规格说明，并按需迭代。
14. **知识归档** —— 扫描已完成设计中的可复用知识，并把候选项展示给用户：

    ```
    📦 知识归档候选项（总体设计完成）

    扫描到以下设计决策值得归档：
    1. [架构决策] 子系统 xxx 承担了 yyy 职责（非显而易见，原设计 zzz 不可行）
    2. [DFX约束] xxx 场景下性能基线为 yyy QPS（来源：压测 / 用户确认）
    3. [外部集成] xxx 接口存在 yyy 限制，通过 zzz 方案适配
    4. [技术选型] 选用 xxx 而非 yyy（原因：zzz 约束）

    请选择要归档的序号（如 1 2 3），或输入"跳过"不归档。
    ```

    若未发现值得归档的内容，仍需告知用户：

    ```
    📦 知识归档候选项（总体设计完成）

    本次设计为常规模式，未发现对团队有通用复用价值的非显而易见决策。

    如有需要补充归档的内容，请现在告知；否则输入"继续"进入模块概要设计阶段。
    ```

    用户确认后，对选中条目调用 `daedalus-knowledge` 执行归档。

15. **⛔ 转换关卡 —— 交接前验证 overall-design-evaluator**：
    - **验证**：`overall-design-evaluator` 是否以等级 ≥ B 通过？或者 `config.evaluators["overall-design"]` 为 `false`？
    - **如果等级 ≥ B 或评估器已跳过**：进入步骤 16。
    - **如果等级 < B**：不得交接。向用户展示问题列表，指出需要补充哪些 init/TRD 章节，然后从步骤 7 返工。
    - **如果无法确认**：不要询问用户。使用 `skills/overall-design-evaluator/agents/evaluator-dispatch-prompt.md`，对 `docs/agent-rules/3-overall-design/output/` 下当前文档重新调度 `overall-design-evaluator`；通过则继续，不通过则返工。

16. **转换到模块概要设计** —— 技术需求设计、总体设计和 OpenAPI 契约均已完成。告知用户：

    > "技术需求设计、总体设计及 OpenAPI 契约已完成，保存在 `docs/agent-rules/3-overall-design/output/YYYY-MM-DD-<project>/`。
    >
    > 下一步：各模块负责人分别调用 `module-design`，该 skill 会引导你选择自己负责的模块并完成模块概要设计。
    >
    > 当前系统包含以下需要模块概要设计的模块/子模块（来自 §5.2/§5.4）：
    > - <module-1>
    > - <module-2>
    > - ..."

    各模块负责人独立调用 `module-design`。该 skill 会按模块处理 init 模板创建和校验，并根据 DESIGN_MODE 路由到完整或轻量概要设计。

## 设计阶段 AI 角色

设计质量来源于在每个阶段有意识地切换到正确模式。

### 作者模式
按团队模板生成设计文档章节。产出文档内容时的默认模式。

### 挑战者模式
在步骤 7（TRD + 方案推荐）以及整个写作过程中都要应用。对每个设计决策：
- 问："这个选择可能先在哪里失败？"
- 问："我在做哪些尚未确认的假设？"
- 问："有没有一个更简单、但被我过早否决的替代方案？"
- 在接受第一个想法之前，强制提出至少一个替代方案。

将未确认的假设内联标记为 `[待确认: reason]`。

### 评审员模式
它定义了子代理评审员（步骤 11）会检查什么。这些也应成为作者在写作过程中始终自查的标准。

**完整性检查（缺少什么）：**
- 系统需求中的每个 REQ-XXX 是否都有对应的设计元素？
- 换一个团队是否能在不提问的前提下实现这份设计？
- 是否存在任何章节描述不足（例如"参考现有实现"但没有足够细节）？

**严谨性检查（有什么，但不该被直接相信）：**
- **伪精确数字**：每个具体数值都必须有可验证来源。否则要么改为范围，要么标记为 `[待验证: 无基准数据，需压测确认]`。
- **跨章节矛盾**：一旦发现一处不一致，要扫描所有相关章节——单点冲突通常隐藏着集群。成组修复。
- **选型理由**：仅有比较表不够。每个评价维度是否有可验证依据？被拒绝方案是否有明确技术理由？
- **客观的章节命名**：设计文档是呈现解决方案，不是自我宣传。"独有的设计创新" → 重构为分析（"领域需求 vs 框架能力差距"）。
- **缺乏依据的接口清单**：实现前通过分析推出的 API 列表，本质上是预测，不是承诺。必须标记为"初步规划（待实现验证）"，并与架构原则分开记录。

**对每个断言的核心问题："证据是什么？"——如果没有证据，要么补充，要么标记为 `[待验证]`。**

### 设计大师模式

**何时调用：** 在方案探索期间（步骤 7）以及撰写关键设计章节期间（步骤 8），使用 `design-master-perspective` 作为额外质量过滤器。设计大师会借助团队积累的设计经验，捕捉挑战者模式和评审员模式可能漏掉的问题。

**如何调用：** 使用 `Skill` 工具调用 `design-master-perspective`。把当前设计决策、方案提案或文档章节提交给设计大师做诊断分析。设计大师将返回：诊断 → 根因 → 建议 → 置信度。

**关键接入点：**

| 设计阶段 | 咨询内容 | 设计大师视角 |
|---|---|---|
| 步骤 7（方案探索） | 各个架构方案 | **风险驱动评估**："每个方案消除了什么风险？又引入了什么风险？" |
| 步骤 8，§2 系统总体架构 | 4+1视图、模块划分、技术选型 | **金字塔拆分检查**："每个模块的粒度是否合适？边界是否清晰？" + 技术选型与 TRD 结论对齐 |
| 步骤 8，§4 系统工作原理 | 系统主流程与核心机制 | **流程完整性检查**："核心流程是否覆盖所有需求场景？异常/边界路径是否考虑完整？" |
| 步骤 8，§5 模块设计说明 | 设计决策、数据架构、模块设计要求 | **易变/不变分离**："变化的部分和稳定核心是否被分开了？" + 数据架构实体归属边界检查 |
| 步骤 8，§6 系统DFX特性设计 | DFX 章节草稿 | **边界纪律**："该 DFX 方案是否正确作用于跨模块边界？" |
| 步骤 8，§9 部署视图 | 部署拓扑与HA策略 | **部署合理性**："部署拓扑是否与物理视图一致？HA策略是否覆盖关键组件？" |
| 步骤 12（质量评估） | 调度评估器之前 | **反模式扫描**：在正式评估前筛查 7 种常见设计反模式 |

**挑战者模式负责在设计过程中阻止坏决策，评审员模式负责发现缺失项和不可信内容，设计大师模式作为第三层过滤器提供团队积累的设计经验。三者都是强制性的。**

## 设计判断 vs 假设

每个设计决策，要么是**锁定判断**，要么是**假设**。

### 锁定判断
至少要由以下之一支撑：
- 知识库文档（如果 `SPEC_DEVELOPER_SERVER_URL` 已配置，则需通过 `daedalus-knowledge` 验证）
- 系统需求文档中的明确表述
- `rules/` 文件中的团队标准
- 当前会话中的用户确认

锁定判断应写成陈述句："系统使用 Redis 进行会话存储。"

### 假设（[待确认]）

总体设计撰写过程中，没有明确依据的设计决策就是假设，必须内联标记为 `[待确认]`，不得写成判断句式。

**标记格式：**
> 设计陈述。**[待确认: 原因——缺少什么信息、为什么无法确定]**

**哪些内容必须标记：**
- 无知识库/需求/标准支撑的设计选择
- 缺少历史数据或压测基线的数值 → `[待确认: 无基准数据，需压测确认]`
- 对现有系统行为的推测 → `[待确认: 需对照代码验证]`
- **禁止**从模板示例值直接搬运，必须替换为合理估计并标记

**清理：**
撰写各章节时持续标记，在增量对齐检查点集中清理——每个检查点扫描已写章节中所有 `[待确认]` 条目，列出清单让用户逐条确认（见"增量对齐" → `[待确认] 清零`）。全部检查点通过后，文档中不得再有未清理的条目。

## 探索设计方案

步骤 7 是与用户的交互式对话，不是模板填空。目标是在 TRD 撰写过程中就总体架构方向与用户达成一致，结论写入 TRD 文档。后续总体设计的 §2.5（技术选型说明）和 §5.1（设计决策记录）直接引用 TRD 的最终结论，不再重复分析过程。

### 如何提出方案

- 提出 2-3 个真正不同、权衡明显不同的方案，而不是同一思路的小变种
- **先给出你推荐的方案**，并解释为什么推荐
- 用对话方式呈现，不要堆成整面文字墙——一次一个方案，然后问
- 全部方案讲完后，必须明确说出："我推荐方案 X，因为……"
- 如果知识库中的某份文档已经对选型形成限制，要明确指出，并解释它如何缩小了选择空间

### 挑战者模式用于方案

对每个待呈现方案，在发给用户前先做挑战者审查：

- "负载上来后最先失败的是什么？"
- "最难测试的部分是什么？"
- "这会给值班/运维带来什么负担？"
- "我在做哪些可能错误的假设？"

任何经不起挑战者审查的方案，都不应呈现给用户，应先替换成更可辩护的方案。

挑战者审查通过后，再应用设计大师诊断：
- **风险扫描**：对每个方案，问"这个方案消除了什么风险？又引入了什么新风险？"
- **边界扫描**："子系统边界在哪里？每个子系统是否只有一个清晰职责？"
- **易变扫描**："当需求演进时，最先变化的部分是什么？它们是否与稳定核心隔离？"
- 当权衡涉及两个以上方案或存在明显 DFX 权衡时，通过 Skill 工具调用 `design-master-perspective`

### DFX 6 个评估维度

当权衡涉及非功能性方面时，可从以下 6 个维度评估：

| 维度 | 关键问题 |
|-----------|-------------|
| **可调试性** | 问题能否快速发现并定位？ |
| **可测试性** | 能否高效、独立地测试？ |
| **可靠性** | 发生故障时能否继续工作？ |
| **可运维性** | 能否在生产环境中稳定运维？ |
| **可扩展性** | 需求增长时能否平滑扩展而不破坏现有行为？ |
| **可复用性** | 组件能否跨子系统或项目复用？ |

并不是每个方案都需要完整覆盖 6 个维度，应聚焦真正区分各方案的维度。

### 连接到 §2.5 / §5.1

步骤 7 中批准的方案，会分别进入设计文档中的 §2.5（选择了什么技术/方案）和 §5.1（该方案如何在设计中落实）。两节只需记录最终选择和理由——选项分析和加权评分已经在 TRD 中完成。按模板表格填写即可。

## 增量对齐

设计文档应分层编写，并在自然决策边界上与用户对齐，而不是一次性倾倒完整草稿。每个检查点都是一个自然的决策边界：如果这里有错误，修复起来成本低。如果跳过检查点在错误的基础上再写 5 章，返工成本就高了。

**不要整份文档写完再统一请用户批准。** 应在有意义的块完成后及时确认。

### 总体设计对齐点

| 检查点 | 覆盖章节 | 需要确认什么 |
|-----------|-----------------|-----------------|
| **架构检查点** | 第 2 章 + 第 4 章（系统总体架构 + 系统工作原理） | 模块划分是否正确？核心流程是否成立？是否缺失模块？ |
| **模块设计检查点** | 第 5 章（模块设计说明） | §5.1 设计决策是否明确？§5.2 模块静态结构是否合理？§5.3 系统数据架构核心实体是否完整？数据归属边界是否正确？§5.4 模块设计要求是否覆盖全部模块？确认后再写第 6 章 DFX。 |
| **DFX 检查点** | 第 6 章（系统DFX特性设计） | 性能目标是否合理？安全方案是否获批？可靠性架构是否成立？ |
| **风险检查点** | 第 8 章（方案风险分析） | 风险识别是否完整？风险等级和影响评估是否合理？应对措施是否可行？ |
| **部署检查点** | 第 9 章（部署视图） | 部署拓扑是否匹配基础设施？HA 策略是否可行？ |
| **完整文档评审** | 全部章节 | 在进入 API 规格说明前完成最终整体评审 |

**每个检查点的强制动作：`[待确认]` 清零**

在每个检查点暂停时，AI 必须扫描当前已写章节中所有 `[待确认]` 条目，列出清单，让用户逐条确认、修正或提供数据：

```
📋 [待确认] 条目清单（§2-§4 已写章节）

§2.2.3 物理视图：假设部署在单机房，无异地容灾。[待确认: 未获取到基础设施信息]
§2.5 技术选型：消息队列暂选 RabbitMQ。[待确认: 团队是否有 Kafka 既有运维能力？]
§4.2.1 主流程：请求超时阈值暂定为 30s。[待确认: 无历史超时数据]

请逐条确认或修正，确认完毕后进入下一段撰写。
```

**清单为空时也要告知：** "✅ 已写章节（§2-§4）无 [待确认] 条目，继续下一段。"

`[待确认]` 条目未清零前，不得进入下一段撰写。

### API 规格说明对齐点

| 检查点 | 需要确认什么 |
|-----------|-----------------|
| **端点列表** | 在写 schema 前先展示所有端点（方法 + 路径 + 一行用途）。确认范围是否正确、是否有遗漏、命名是否规范 |
| **完整规格评审** | schema 与错误码补齐后的整体评审 |

### 如何表述检查点

完成某个检查点后，可以这样写：

> "第 2-3 章已完成，涵盖了 [概要]。继续之前需要确认几件事：
> 1. [具体问题 1]
> 2. [具体问题 2]
> 这些是否正确，还是需要我调整？"

检查点消息应简洁，只聚焦用户需要确认的关键决策。

## 知识库发现

**查询前，必须先通过读取 `cospowers.config.json` 判断适用模式：**

```
1. 知识库查找（补充上下文，使用最佳可用来源）：
   - config.kb.skill 已设置且环境中可用？ → 调用该 skill
   - 否则 kb-query 可用？ → 使用 kb-query
   - 否则 config.kb.localPath 存在？（默认：doc/kb/） → Grep/Read 本地知识库文件
   - 否则 docs/kb/ 存在？ → Grep/Read docs/kb/ 文件
   - 否则 → 跳过知识库查找

2. 代码优先：始终直接在代码库上做 Glob + Grep + Read
```

**知识库指引方向，代码验证真相。** 无论哪种模式，始终对照实际代码验证发现（步骤 C）。

### 步骤 A：理解架构

| 模式 | 操作 |
|------|--------|
| **kb-skill / kb-query** | 查询系统架构、技术栈、跨产品共享模块 |
| **local-kb** | 读取 `<config.kb.localPath>` 下的架构/模块概览文件 |
| **代码优先** | Glob 入口点、路由、主服务文件；读取顶层 README 和目录结构 |

目标：理解子系统拆分、各模块技术栈以及共享组件。

### 步骤 B：深入受影响模块

| 模式 | 操作 |
|------|--------|
| **kb-skill / kb-query** | 查询各受影响模块的 API、依赖、排障知识 |
| **local-kb** | 用模块名 Grep `<config.kb.localPath>` |
| **代码优先** | 读取受影响模块的 handler/service/router 文件，并 Grep 接口定义 |

每个模块至少要找出：
- 内部组件关系、设计模式
- 现有 API（请求/响应格式）
- 依赖关系（它依赖谁，谁依赖它）
- 已知故障模式

### 步骤 C：代码验证

无论哪种模式，都必须验证：

1. **Glob** 查找知识库或 README 提到的文件
2. **Grep** 搜索关键函数、类、接口定义
3. **Read** 验证模块边界、API 签名、数据模型
4. **当知识库与代码不一致时，以代码为准**

### 知识库发现如何输入设计决策

| 知识库发现 | 输入到哪个设计元素 |
|----------------------|-------------------------------|
| 系统架构图 | 总体设计第 2 章 —— 尊重既有子系统边界，以扩展为主，不轻易替换 |
| 各模块技术栈 | 总体设计 §2.5 —— 技术选型需与现有技术栈对齐 |
| 现有 API | OpenAPI 规格说明 —— 优先扩展已有端点，保持向后兼容 |
| 模块依赖 | 总体设计 §2.4 —— 依赖影响分析 |
| 已知故障模式 | 总体设计 §6.2 —— 在设计中响应已知可靠性问题 |

## 历史设计处理

如果该系统已有上一版设计文档，则对每个主要章节、子系统和接口打上变更标签：

| 标签 | 含义 |
|-----|---------|
| `[延续]` | 与上一版设计一致，无变化 |
| `[新增]` | 本轮新增，之前无对应项 |
| `[变更-来自§X.X]` | 相比上一版设计第 X.X 节发生修改——描述变更内容 |
| `[废弃]` | 上一版存在，本轮移除——解释原因 |

**修订记录要求（ch11-变更控制与修订记录 变更控制）：**
每条修订记录都必须说明变更内容及其原因，不能只列版本号：

```
| V2.0 | 张三 | 2026-04-09 | 新增用户权限子系统（来自REQ-012），废弃旧ACL实现（安全合规要求）|
```

"更新设计"或"按需求修改"都不是可接受的修订描述。

## OpenAPI 3.0 规格说明要求

### 输出格式

API/接口规格说明（步骤 9）**必须**产出为 **OpenAPI 3.0 YAML**，不能是 JSON，也不能是 Markdown 表格。文件结构遵循 `config.templates["openapi"]`（默认：`templates/openapi-template.yaml`）。输出文件扩展名为 `.yaml`。

### 参数约束

对于存在正式 API 类型系统的项目，所有字段约束都应遵循项目约定。撰写 OpenAPI schema 字段时：

1. 确定字段对应的类型约束（字符串长度、数值范围、枚举值、格式等）
2. 将这些约束映射为 OpenAPI 3.0 schema 属性（`minLength`、`maxLength`、`minimum`、`maximum`、`enum`、`format`、`pattern`）
3. 如果项目使用自定义类型系统（如 `x-type`），应附加扩展属性记录原始类型，便于后端参考

## 3 条设计红线

这些都是**强制要求**，在标记设计完成前必须逐项验证：

### 红线 1：章节完整性

**设计文档中任何章节都不得为空。** 每个章节都必须包含实质性内容。如果章节不适用，必须明确写出"不适用"及其原因。此规则在模板顶部写明。

### 红线 2：可靠性章节

设计**必须**包含可靠性相关内容，覆盖：
- 功能承载组件的容错分析
- 资源使用分析（不得存在失控资源消耗）
- 业务流程可靠性（每条业务流都必须可靠）
- 分析要基于容错思维，而不是代码缺陷列表

### 红线 3：安全章节

设计**必须**包含安全相关内容，覆盖：
- 安全基线机制集成
- 威胁建模分析（如安全经理要求）
- 基于风险识别结果的安全设计
- 使用前组件的版本合规性与漏洞状态

## DFX 维度

### 总体设计 DFX（模板第 6 章）

总体设计模板要求覆盖 11 个 DFX 维度，并且这些内容关注的是**跨子系统的架构决策**（不是单一子系统实现细节）：

| 维度 | 模板章节 | 关键内容 |
|---|---|---|
| **安全性（Security）** | 6.1 | 安全架构、安全域划分、数据传输安全 |
| **可靠性（Reliability）** | 6.2 | HA 架构、故障检测/恢复机制、数据可靠性 |
| **性能（Performance）** | 6.3 | 性能优化策略、瓶颈分析 |
| **可扩展性（Scalability）** | 6.4 | 水平与垂直扩展能力 |
| **可运维性（Operability）** | 6.5 | 监控架构、日志架构、部署架构 |
| **兼容性（Compatibility）** | 6.6 | API 版本管理、升级/迁移策略、跨子系统兼容性约束 |
| **可测试性（Testability）** | 6.7 | 测试环境架构、跨子系统集成策略、E2E 测试设计 |
| **可调试性（Debuggability）** | 6.8 | 诊断可观测性、链路追踪、排障挂钩 |
| **可复用性（Reusability）** | 6.9 | 可复用模块、公共代码、技术规范沉淀 |
| **系统隐私（Privacy）** | 6.10 | 隐私边界与敏感数据处理 |
| **软件质量及效能改进（Quality/Efficiency）** | 6.11 | 工程质量与交付效率改进 |

**层次定位规则**（来自模板）：总体设计 DFX 仅覆盖跨子系统架构方案。三份文档的 DFX 分工为：
- 系统需求文档第 4 章：定义 DFX **指标与需求**（例如 99.9% 可用性、RTO<5min）
- 总体设计第 6 章：定义满足这些指标的**架构解决方案**（例如安全域设计、HA 拓扑、监控系统选型）
- 模块概要设计 Ch.5：定义各模块的**具体实现**（例如健康检查端点、重试逻辑、日志格式）——由 `module-design-spec` 负责
