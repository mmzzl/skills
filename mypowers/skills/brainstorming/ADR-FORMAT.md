# ADR Format

ADR 存放在 `docs/agent-rules/specs/adr/` 目录，使用顺序编号命名：`0001-slug.md`、`0002-slug.md`，依次递增。

创建 `adr/` 目录时惰性处理——只有在第一个 ADR 需要写入时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么、我们决定了什么、为什么这样决定。}
```

就这些。一个 ADR 可以只有一段话。价值在于记录"做了这个决定"以及"为什么"，而不是填满各个章节。

## 可选章节

只在真正有价值时才加。大多数 ADR 不需要这些。

- **Status**（frontmatter）：`proposed | accepted | deprecated | superseded by ADR-NNNN`——在决策被重新审视时有用
- **Considered Options**：只有当被否决的备选方案值得记录时才加
- **Consequences**：只有当非显而易见的下游影响需要被点明时才加

## 编号规则

扫描 `docs/agent-rules/specs/adr/` 目录，找到最大的现有编号，加一。

## 三条创建门控

以下三条**必须全部满足**才创建 ADR，缺任何一条就跳过：

1. **难以逆转** — 事后改变主意的代价是实质性的
2. **没有上下文会令人困惑** — 未来的读者看到代码会纳闷"他们为什么这么做？"
3. **真实取舍的结果** — 有真实的备选方案，选了其中一个是有具体原因的

如果决策容易撤销，就跳过——直接撤销就好。如果不会让人困惑，也没人会问为什么。如果没有真正的备选方案，那只是"做了显而易见的事"，不需要记录。

## 什么样的决策符合条件

- **架构形态**："我们使用 monorepo。" "写模型是事件溯源，读模型投影到 Postgres。"
- **子系统间集成模式**："Ordering 和 Billing 通过领域事件通信，而非同步 HTTP。"
- **带来锁定效应的技术选型**：数据库、消息总线、认证方案、部署目标。不是每个库——只有那些换掉需要一个季度的。
- **边界与范围决策**："客户数据由 Customer 子系统拥有，其他子系统只通过 ID 引用。" 明确的"不做"和"做"同样有价值。
- **蓄意偏离显而易见路径**："我们用原生 SQL 而不是 ORM，因为 X。" 任何合理读者会假设相反做法的情况——避免下一个工程师把刻意的决策当成 bug 修掉。
- **代码中不可见的约束**："我们不能用 AWS，因为合规要求。" "响应时间必须低于 200ms，因为合作方 API 合同。"
- **否决原因不显而易见的备选方案**：如果考虑过 GraphQL 但因为微妙原因选了 REST，就记录下来——否则六个月后有人又会提 GraphQL。
