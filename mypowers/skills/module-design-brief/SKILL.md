# 模块概要设计（轻量）

**Skill 标识**: `module-design-brief`

其他 skill 通过 `module-design-brief` 引用本 skill。

## 概述

当模块概要设计不需要完整覆盖全部 9 个章节时使用本 skill。仅完整撰写 ch04-内部设计和 database-design（涉及持久化时），其余章节生成骨架标注"不涉及"。

**设计原则：只减内容，不减流程。** 保留完整的质量管线（挑战者→评审员→设计大师→子代理评审→知识归档→writing-plans 交接），流程步骤与完整概设一致，区别仅在于大部分章节产出为骨架而非完整内容。

**输入：** 总体设计文档（由 `overall-design-spec` 产出）
**输出：** 模块概要设计文档包（先产出 10 个独立文件用于评审，评审通过后合并为 1 个 `module-design-brief.md` 文件）

**核心原则：** 无空章节，ch04 必须有 Mermaid 图，数据库表使用 SQL DDL，未确认假设标记 `[待确认]`，产出可评审。

## 扩展点

开始之前，从插件根目录（此 skill 基目录向上 2 级——即 skill 加载时显示的"此 skill 的基目录"）读取 `cospowers.config.json`。无需回退——配置文件始终包含有效默认值。

| 配置字段 | 用途 |
|---|---|
| `config.templates["module-design"]` | 模块概要设计文档的模板文件路径 |

本 skill 不使用 evaluator、design-review 和 dfx 规则配置。

## 何时使用

- 用户只需填写内部设计（ch04）和数据库设计，其他章节不需要展开
- 模块接口已在总体设计/OpenAPI 中明确定义，不需要重复描述
- 不需要 DFX 设计、自测用例、部署方案等完整模块概要设计内容
- 用户明确说"只写内部设计""只要 ch04""其他写无"等

**以下情况不要使用：**
- 需要完整的模块概要设计文档（请改用 `module-design-spec`）
- 需要模块方案选型（请改用 `module-design-spec`）
- 总体设计尚未完成（请先使用 `overall-design-spec`）

## 检查清单

你必须为以下每一项创建任务，并按顺序完成：

1. **加载总体设计并选择目标模块** —— 扫描 `docs/agent-rules/3-overall-design/output/` 中前缀为 `YYYY-MM-DD-` 的子目录；取最新的（按降序排列）。

   - 读取 `index.md` 了解产品、迭代和章节目录
   - 从以下章节提取目标模块上下文：
     - §2.3 模块划分 / §2.4 模块依赖关系 —— 模块职责和依赖
     - §5.2 模块静态结构 —— 模块层级位置
     - §5.3 系统数据架构 —— 与该模块相关的全局实体
     - §5.4 模块设计要求 —— 设计目标和约束
   - **加载 OpenAPI 规格说明：** 从同一目录读取 `<project>-openapi.yaml`，提取与目标模块相关的端点定义（方法、路径、请求/响应 schema）和错误码定义。ch03 骨架章节引用 OpenAPI 中的接口定义。
   - 向用户提交总设中的模块列表：

   > "总体设计包含以下模块（来自 §2.3）：
   > - <module-1>
   > - <module-2>
   > - ...
   >
   > 你负责哪个模块的内部设计？"

   **本次调用范围限定为一个模块**。

   - **查询产品知识库（轻量）** —— 在选择目标模块之后、收集 init 模板之前，使用模块名调用知识库（仅查 3 项）：
     - 模块业务领域与上下游依赖 → 预填 init §4 业务规则
     - 现有 API 清单（防止 ch03 骨架与现有 API 不一致）
     - 现有数据模型（防止 ch04 §4.5.1 DDL 与现有冲突）

     **四级降级检测**：`config.kb.skill` → `kb-query` → `config.kb.localPath` → 代码优先（详见下方"知识库发现（轻量）"章节）。**知识库指引方向，代码验证真相**——在 3a 读取代码时以代码为准。

2. **收集实现上下文** —— 通过 init 模板收集模块负责人掌握的、总设中没有包含的实现约束。

   **Init 模板章节 → 设计步骤映射：**

   | 章节 | 提取内容 | 输入到 |
   |---|---|---|
   | §1 模块基本信息 | 模块确认、仓库/服务归属、功能范围、依赖关系 | ch04 §4.1 子模块划分；ch04 §4.2 架构影响 |
   | §2 模块内部结构图 | 内部组件/子模块图 | ch04 §4.1 子模块划分；对照代码验证 |
   | §3 技术债务和遗留问题 | 已知 bug、冻结代码、遗留约束 | ch04 §4.6 异常处理、§4.7 设计要点 |
   | §4 业务规则和领域知识 | 代码中不可见的领域规则 | ch04 §4.3 流程设计、§4.4 核心算法 |
   | §5 涉及服务的本地代码路径 | 涉及服务的本地代码路径 | 代码验证；实现上下文 |

   - 从 skill 的基目录读取 `templates/init-module-design-brief-template.md`
   - 预填已知信息（模块名称、总设中的模块设计要求、OpenAPI 中的 API 候选）
   - 将预填内容写入 `docs/agent-rules/4-module-design/input/init-{module-name}-brief.md`
   - 告知用户：

   > "我已为「{module-name}」生成背景情报简报，保存在 `docs/agent-rules/4-module-design/input/init-{module-name}-brief.md`，请在 IDE 中打开填写。
   > **§1 模块基本信息为必填项**；§2 内部架构图强烈建议提供；§4 技术债务务必如实填写；§8 代码路径建议填写。填写完成后告知我。"

   - **等待用户确认后再继续。**

   **⛔ 硬关卡 —— 步骤 2a：Init 完整性校验**

   | 章节 | 是否阻塞？ | 充分条件 |
   |---|---|---|
   | §1 模块基本信息 | **是** | 模块名称已确认，服务/仓库归属已说明，或明确未知并说明原因 |
   | §2 模块内部结构图 | 否 | 图已提供；若无，注明将从代码推断模块拆分 |
   | §3 技术债务 | 否 | 列出已知问题或明确写"无已知债务"——留空可接受 |
   | §4 业务规则 | 否 | 领域规则具体且可测试 |
   | §5 代码路径 | 否（软阻塞） | 路径在磁盘上存在（Glob 验证）；若未找到，告知用户 |

   如果任何阻塞项失败：告知用户具体需要什么，等待更新，重新读取并重新验证。**在阻塞项通过之前不要开始撰写。**

3. **撰写 ch04 内部设计 + 骨架章节**

   **3a. 读取代码**（如果步骤 2 提供了代码路径）
   - Glob 入口文件、核心模块文件
   - Grep 关键函数、类、接口定义
   - Read 关键文件理解现有实现
   - 当代码与总设描述不一致时，以代码为准，不一致处标记 `[待确认: 代码与总设不一致，需确认]`

   **3b. 撰写 ch04-内部设计.md**，遵循 `templates/design/module-design/ch04-内部设计.md` 模板：

   | 子节 | 内容 | AI 可读性要求 |
   |------|------|-------------|
   | §4.1 子模块划分 | 子模块清单表 + Mermaid 依赖图（§4.1.2） | Mermaid `flowchart` 或 `graph`，不用 ASCII |
   | §4.2 对总体架构的影响 | 影响分类表（无影响 / 轻微影响 / 重大影响） | — |
   | §4.3 流程设计 | 每个流程：Mermaid `sequenceDiagram`（含 `alt`/`else` 异常分支）+ 步骤表（标注 `[内部]`/`[外部]`，异常处理列含错误码和回滚动作）+ 函数列表 | 外部调用标注接口路径；异常分支覆盖参数校验失败/认证失败/外部依赖不可用/业务逻辑异常/部分成功需回滚 |
   | §4.4 核心算法与公共框架 | 算法伪代码 + 性能分析表（时间/空间复杂度最佳/平均/最差）+ 框架机制 | 仅核心算法可用伪代码，其他节禁止代码片段 |
   | §4.5 数据结构设计 | §4.5.1 数据库表（SQL DDL，含 COMMENT + 索引注解 + 表归属仓库）；§4.5.2 内存数据结构；§4.5.3 全局数据结构；§4.5.4 配置文件（路径/格式/字段类型/默认值/取值范围/是否需重启） | DDL 不用 Markdown 表格，枚举字段列合法值；配置文件标注"修改后是否需重启" |
   | §4.6 异常场景处理 | 异常场景清单表（场景/触发条件/影响范围/处理策略/恢复方式）+ 降级策略表 | — |
   | §4.7 设计要点检视 | 可维护性/可调试性/可测试性/自动化测试支持/可扩展性/稳定性/工作量评估 | 自检清单，逐项评估 |

   **应用三种设计角色：**

   | 角色 | 应用方式 |
   |------|---------|
   | **挑战者模式** | 对每个设计决策追问"这个选择可能在哪里失败""我做了哪些未确认的假设""有没有更简单但被过早否决的替代方案"。将未确认的假设内联标记为 `[待确认: reason]`。 |
   | **评审员模式** | 自查完整性（每个流程是否覆盖异常分支？换一人能否不提问就实现？）和严谨性（数值是否有可验证来源？步骤表函数名是否具体？跨子节是否一致？）。 |
   | **设计大师模式** | 通过 `Skill` 工具调用 `design-master-perspective`，2 个接入点：撰写 §4.1 子模块拆分后 → **金字塔分解检查**（"每个子模块的粒度是否合适？边界是否清晰？"）；撰写 §4.6 异常处理后 → **错误截断启发式**（"每个错误是否在最早可能的位置被检测？错误传播是否有界？"）。 |

   **3c. 撰写 database-design.md**（当模块涉及持久化存储时）

   使用 `skills/module-design-spec/agents/data-model-designer.md` 创建子代理，遵循 `templates/design/module-design/数据库设计模板.md` 模板撰写，写入 `docs/agent-rules/4-module-design/output/YYYY-MM-DD-<project>/<module>/database-design.md`。

   **§4.5.1 与 database-design.md 的关系：**
   - §4.5.1（ch04 内部设计的数据结构子节）服务于**模块级设计决策**——用 SQL DDL 描述关键表，说明表与模块功能的关系、关键索引的查询用途
   - database-design.md 服务于**完整的数据库实施**——包含全部表结构、ER 图、命名规范、分区策略、备份恢复、数据安全等，是 DBA 和开发团队的独立参考文件
   - **写入规则：** database-design.md 包含完整 DDL，§4.5.1 从 database-design.md 中提取与本模块最相关的关键表（3-5 张），标注"完整建表脚本见 database-design.md"。两处 DDL 必须一致，不一致时以 database-design.md 为准
   - 如果模块不涉及持久化存储，database-design.md 不创建，§4.5.1 标注"不涉及"

   **3d. 生成骨架章节**，遵循主模板全局约束（"所有章节不得为空；如不涉及须写明'不涉及'及原因"）：

   | 文件 | 内容 |
   |------|------|
   | `index.md` | 文档元数据表 + 高密度概要（一句话说明本次仅涉及内部设计）+ 章节目录（每章标注"骨架"或"完整"） |
   | `ch02-模块职责和边界.md` | 从总设 §2.3/§5.2/§5.4 提取模块职责概述（3-5 行），末尾标注"本次仅填写内部设计，职责边界详见总体设计" |
   | `ch03-对外接口.md` | 标注"接口契约见 OpenAPI 规格说明，本次不展开" |
   | `ch05-DFX特性设计.md` | 9 个 DFX 维度逐一标注"不涉及，本次仅填写内部设计" |
   | `ch06-自测用例.md` | 标注"不涉及，本次仅填写内部设计" |
   | `ch07-部署与发布.md` | 标注"不涉及，本次仅填写内部设计" |
   | `ch08-总结与变更控制.md` | 标注"不涉及，本次仅填写内部设计" |
   | `ch09-附录.md` | 术语表（如有模块特有术语）+ 参考资料（如有）；若无，标注"不涉及" |

   输出目录：`docs/agent-rules/4-module-design/output/YYYY-MM-DD-<project>/<module>/`

4. **跨章节一致性检查** —— 验证：
   - ch04 中对总体设计的引用（模块职责、依赖关系、数据架构）与总设相关章节一致
   - ch04 §4.5.1 DDL 与 database-design.md 表结构一致（如适用）
   - 术语在 ch04 各子节间一致使用

5. **子代理评审** —— 使用 `skills/module-design-brief/agents/evaluator-dispatch-prompt.md` 调度子代理评审员，评审整个模块设计目录（ch04 内部设计 + database-design 完整内容 + 骨架章节合规性）。修复发现的问题后再进入步骤 6。

   评审结果需达到等级 B（≥80）方可继续。若 < B，修复 Error/Critical 问题后重新调度，最多 2 轮；若仍 < B，向用户展示剩余问题然后继续。

6. **文件合并** —— 评审通过（等级 ≥ B）后，将独立章节文件合并为单一文档。

   按以下顺序合并所有章节文件：

   ```
   index.md → ch02-模块职责和边界.md → ch03-对外接口.md → ch04-内部设计.md → ch05-DFX特性设计.md → ch06-自测用例.md → ch07-部署与发布.md → ch08-总结与变更控制.md → ch09-附录.md → database-design.md（如存在）
   ```

   合并后输出至：`docs/agent-rules/4-module-design/output/YYYY-MM-DD-<project>/<module>/module-design-brief.md`

   **合并规则：**
   - 每个原文件内容作为一个一级章节（`#` 标题），章节标题与文件名对应（去掉 `.md` 后缀和 `ch0X-` 前缀）
   - 保留原文件中的所有内容、标记、图表、Mermaid 代码块
   - `[待确认]` 标记原样保留，进入用户评审阶段（步骤 7）集中清零
   - 合并完成后必须删除原始独立文件

   **合并后验证：** 确认合并文件包含所有章节、Mermaid 代码块完整闭合、无内容截断或重复。

7. **用户评审** —— 展示合并后的 `module-design-brief.md`（重点审阅 ch04 和 database-design 章节）。

   **`[待确认]` 清零：** 扫描合并文件 `module-design-brief.md` 中所有 `[待确认]` 条目，列出清单让用户逐条确认、修正或提供数据：

   ```
   📋 [待确认] 条目清单

   ch04 §4.1：子模块划分假设 xxx 和 yyy 可独立部署。[待确认: 未获取到部署单元信息]
   ch04 §4.3.1：请求超时阈值暂定为 30s。[待确认: 无历史超时数据]
   ch04 §4.6：降级策略暂定返回缓存数据。[待确认: 是否有现成缓存方案？]

   请逐条确认或修正，确认完毕后完成用户评审。
   ```

   **清单为空时也要告知：** "✅ `module-design-brief.md` 无 [待确认] 条目，设计已确认。"

   `[待确认]` 条目全部清零后，用户评审即完成。

8. **知识归档** —— 扫描合并后的 `module-design-brief.md` 中可复用知识候选项：

   - **架构决策**：非显而易见的子模块拆分模式、设计选择及其原因
   - **异常处理模式**：有通用价值的降级策略、错误传播机制
   - **算法设计**：可跨项目复用的核心算法

   向用户提交候选项：

   ```
   📦 知识归档候选项（内部设计完成）

   扫描到以下内容值得归档：
   1. [架构决策] xxx 模块采用 yyy 拆分子模块（原因：zzz）
   2. [异常处理] xxx 场景的降级策略...

   请选择要归档的序号（如 1 2），或输入"跳过"不归档。
   ```

   若未发现候选项，告知用户：

   ```
   📦 知识归档候选项（内部设计完成）

   未发现值得归档的内容。

   如有需要补充归档的内容，请现在告知；否则输入"继续"。
   ```

   用户确认后，对选中条目调用 `daedalus-knowledge` 执行归档。

9. **转换到实现** —— 内部设计完成。告知用户：

   > "`<module>` 模块内部设计已完成，保存在 `docs/agent-rules/4-module-design/output/YYYY-MM-DD-<project>/<module>/`。
   >
   > 合并文档：`module-design-brief.md`
   >
   > 如需进入实现阶段，可以调用 `writing-plans` 生成实现计划。"

## 知识库发现（轻量）

**简版概设仅做 3 项知识库查询：业务领域、现有 API、现有数据模型。** 其他维度（故障模式、技术债、上下游依赖）由 Step 2 init 模板负责人填写和 Step 3a 代码读取覆盖，不在 KB 阶段单独查询。

**四级降级检测**（读取 `cospowers.config.json`）：

```
1. 知识库查找（补充上下文，使用最佳可用来源）：
   - config.kb.skill 已设置且环境中该 skill 可用？→ 调用该 skill
   - 否则 kb-query 在环境中可用？→ 使用 kb-query
   - 否则 config.kb.localPath 存在？（默认：doc/kb/）→ Grep/Read 本地知识库文件
   - 否则 docs/kb/ 存在？→ Grep/Read docs/kb/ 文件
   - 否则 → 跳过知识库查找

2. 代码优先：始终直接在代码库上 Glob + Grep + Read
```

**知识库指引方向，代码验证真相。** KB 结果对照 ch04 §4.5.1 DDL 验证一致性；当 KB 与代码不一致时以代码为准。

| 知识库发现 | 输入到 |
|---|---|
| 模块业务领域 / 上下游依赖 | init §4 业务规则（预填） |
| 现有 API 清单 | ch03 骨架（与现有 API 保持一致） |
| 现有数据模型 / 表结构 | ch04 §4.5.1 DDL（避免与现有 schema 冲突） |

## 关键约束

### 红线 1：章节完整性

**所有章节不得为空。** 骨架章节必须明确写出"不涉及"及原因。此规则在主模板顶部写明。

### 强制模板元素

- ch04 §4.1.2 子模块依赖图必须用 Mermaid 呈现
- ch04 §4.3 每个流程必须有 Mermaid `sequenceDiagram`（含异常分支）+ 步骤表
- ch04 §4.5.1 数据库表必须用 SQL DDL 格式，含 `COMMENT` 和索引注解
- 步骤表标注 `[内部]`/`[外部]`，外部调用标注接口路径

### 设计判断 vs 假设

每个设计决策要么是锁定判断（有总设/代码/标准/用户确认支撑），要么是假设（必须标记 `[待确认: reason]`）。绝不能把假设当作判断陈述。所有 `[待确认]` 条目在步骤 6 用户评审时集中清零。
