# Generate IPD Story

**Skill 标识**: `generate-ipd-story`

其他 skill 通过 `generate-ipd-story` 引用本 skill。

## 概述

基于 `system-requirement-analysis` 产出的系统需求目录，生成 IPD 格式的 AI 需求文档（Epic → Feature → Story → Tech）。将 REQ-XXX 条目映射为 Story，将业务场景映射为 Feature，技术需求映射为 Tech（含系统级和服务级），使用 `templates/ipd-story-template.md` 模板。

**核心原则：** 不得凭空扩展范围。仅映射输入文档中已有的内容。【技术思路】和【工作量评估】始终留给人工填写。

**生成策略：两阶段写入（框架 → 细节）。** 先写骨架结构让用户确认，避免框架错误导致细节返工。框架确认后再填充完整内容。

## 使用时机

- `system-requirement-analysis` 完成输出目录写入后（在 Step 2.3 自动 dispatch 本 skill）
- 用户明确要求从已有系统需求目录生成 Epic/Feature/Story 文档

**不适用场景：**
- 系统需求文档尚未完成——请先完成 `system-requirement-analysis`
- 输入为原始产品需求且无结构化文档——请使用 `requirement-analysis`

## 执行步骤

每个步骤必须创建对应任务，按顺序完成。

### 步骤 0：定位系统需求目录

1. 若已提供 `sysreq_dir`：直接使用该目录。
2. 若未提供：扫描 `docs/agent-rules/2-system-requirements/output/` 下以 `YYYY-MM-DD-` 为前缀的子目录，按日期降序取最新一个。
3. 读取 `index.md`，获取项目名称、REQ 数量、优先级分布。
4. 读取以下章节文件：
   - `ch02-关键应用场景.md` — Feature 和 Epic 的主要分组依据
   - `ch03-功能性需求.md` — 所有 REQ 条目（含优先级、描述、验收条件）
   - `ch04-非功能性需求DFX.md` — DFX 指标
   - `ch07-设计任务书.md` — 完整 REQ 清单

**⛔ 前置门控：** 若 ch02、ch03、ch04、ch07 任一缺失或为空，停止并告知：
> "系统需求文档不完整——未找到 [缺失文件名]。请先完成 `system-requirement-analysis`。"

### 步骤 1：加载模板

读取 `templates/ipd-story-template.md`，内化以下内容：
- Epic → Feature → Story → Tech 完整章节结构
- 8 项 QS 强制通过项（QS-001 至 QS-008）
- 各层级必填章节（Epic：8 节，Feature：6 节，Story：所有必填字段，Tech：系统级+服务级）
- 全局约束表（不得使用占位文本或模糊措辞）

### 步骤 2：规划 E/F/S/T 结构

写入前，向用户展示映射方案并请确认：

- **Epic**：每个主要业务主题一个
- **Feature**：每个来自 ch02 的独立业务场景一个，承载该场景下的 REQ 条目
- **Story**：每个 REQ 条目一个；若一个 REQ 涵盖多个独立用户操作，可拆分
- **Tech**：每个 Story 下可能有 1~N 个 Tech（根据技术复杂度拆分），分两类：
  - **系统级 Tech**：跨服务调用、架构设计、数据管道等涉及多模块的技术需求
  - **服务级 Tech**：具体 API 实现、数据库操作、业务逻辑等单个服务内的技术需求

**命名必须遵循以下规范**（步骤 4 写入时严格按此格式）：

| 层级 | 格式 | 示例 |
|------|------|------|
| Epic | `Epic{N}-{名称}` | `Epic1-日志审计` |
| Feature | `Feature{N}.{N}-{名称}` | `Feature1.3-审计告警策略` |
| Story | `Story{N}.{N}.{N}-{名称}` | `Story1.5.1-告警规则管理` |
| 系统级 Tech | `Tech-系统级-{名称}` | `Tech-系统级-告警事件总线` |
| 服务级 Tech | `Tech-服务级-{名称}` | `Tech-服务级-告警规则CRUD接口` |

**以树形结构展示方案（含命名）：**
```
Epic1-{主题}
  Feature1.1-{名称}（来源：ch02 场景）
    Story1.1.1-{用户操作}（来源：REQ-XXX）
      Tech-系统级-{技术标题}
        Tech-服务级-{技术标题}.md
    Story1.1.2-...
  Feature1.2-...
```

**交互模式**（用户直接调用）：等待用户确认，按需修改后再进入步骤 3a。

**后台 dispatch 模式**（由 `system-requirement-analysis` 调用）：跳过步骤 2 和 3a 的用户确认，直接按规划进入步骤 3b。

### 步骤 3a：生成框架内容（第一层——骨架）

仅生成每层的骨架字段，不填充场景分析、AC 条件、DFX、Tech 细节等。目标是让用户确认结构正确。

**Epic 框架（只写以下字段）：**
- 【标题】从 ch02 业务主题提取
- 【关键价值点】一句话概要
- 【包含 Feature】列出所有 Feature 标题
- 【质量需求】P0/P1 条目编号列表（不写详细描述）
- 【跨团队协作】标注"涉及/不涉及"

**Feature 框架（只写以下字段）：**
- 【标题】从 ch02 场景名称提取
- 【特性需求分类】分类 + 一句话理由
- 【包含 Story】列出所有 Story 标题
- 【非功能需求约束 DFX】只列 DFX 维度名称，不写具体指标

**Story 框架（只写以下字段）：**
- 【标题】从 REQ 描述提取
- 【用户故事】"作为…我想要…以便…"一句话
- 【场景说明】痛点一句话 + `来源：REQ-XXX`
- 交付标记：🔴（P0）、🟡（P1）、🟢（P2-P3）

**Tech 框架（只写以下字段）：**
- 【技术标题】从 Story 的技术复杂度判断是否需要拆分系统级和服务级
- 【Tech 类型】标注 `[系统级]` 或 `[服务级]`
- 【一句话范围】该 Tech 要解决什么技术问题

**框架输出样例：**
```
## Feature1.1-资源管理

【特性需求分类】基本可用——用户登录后首先看到资源列表，是最基础的操作入口

【包含 Story】
- Story1.1.1-查看资源列表（REQ-001）🔴
  - Tech-系统级-列表分页与缓存设计
- Story1.1.2-创建新资源（REQ-002）🔴
  - Tech-系统级-资源创建异步流程设计
    - Tech-服务级-资源创建服务实现
```

**交互模式**：展示框架给用户，等待确认或修改意见。
**后台 dispatch 模式**：跳过确认，直接进入步骤 3b。

---

### 步骤 3b：填充完整细节（第二层——逐项展开）

框架确认后，逐层填充所有必填字段。按以下顺序填充，每填充完一层可让用户确认：

#### 3b.1 填充 Epic 完整内容

补全：
- 【价值实现程度】量化目标
- 【用户需求设计结论】7 个子项齐全（价值规划、目标用户与旅程、核心痛点、设计策略、Playbook、竞品对比、关键注意事项）
- 【客户验证结论】≥ 3 行（QS-001）——无来源数据时标注 `[待填写]`
- 【质量需求】从 ch03 优先级提取 P0/P1 条目
- 【跨团队协作】从 ch06 跨团队说明提取；无则填"不涉及"
- 【责任人】= "待填写"

#### 3b.2 填充 Feature 完整内容

补全：
- 【关联关键价值点】关联 Epic 价值点
- 【功能级 Playbook】完整场景描述
- 【场景建模】三层分解表（场景复杂时填写，简单时填"不涉及"）
- 【特性验收条件】整体验收标准
- 【非功能需求约束 DFX】（QS-004）——来源 ch04，逐维度填写具体指标
- 【关键交互说明】如有则填写
- 【责任人】= "待填写"

#### 3b.3 填充 Story 完整内容（核心）

补全每个 Story：

**【A/C 验收条件】一、功能性验收条件**

参照 SDD 场景分析深度，每个 Story 必须包含：

1. **业务规则与约束**：该 Story 满足的约束条件

2. **实例化场景（正常路径 + 异常路径 ≥1 + 安全场景）**：

```
正常场景：<场景名称>
  Given（前置条件）：<系统状态、用户状态、数据状态>
  When（行为）：<用户执行的具体操作，分步骤>
  Then（结果）：<系统返回、状态更新、用户可见反馈>

异常场景：<场景名称>
  Given（前置条件）：
  When（行为）：
  Then（结果）：

安全场景：权限越界 / 敏感数据访问 / 注入防护
  Given（前置条件）：
  When（行为）：
  Then（结果）：
```

3. **边界条件**（按需）：空值、最大值、并发等

**【期望效果】** 用户可量化的期望结果

**【需求规格约束】** 🔲 按需填写

**【操作步骤】** 详细描述用户操作步骤

**【A/C 验收条件】二、非功能性验收条件**

- 【产品需求】关联场景、性能指标、稳定性要求——从 ch04 DFX 逐项映射
- 【安全需求】明确填写（QS-007）——不得为空
- 【海外合规】"不涉及"（除非来源文档明确说明涉及海外范围，QS-006）

**【依赖与风险】⭐**
- 需求关联性、内部功能依赖
- 技术风险、测试风险
- 跨团队协作

**【技术思路】** = "待开发负责人填写"
**【工作量评估】** = "待评估"
**【责任人】** = "待填写"

**禁用词：** "合理 / 适当 / 较好 / 尽量 / 性能好 / 适当考虑"

#### 3b.4 填充 Tech 完整内容

每个 Story 下可能有 0~N 个 Tech，分类填充：

**Tech 判断原则：**
- **系统级 Tech**：涉及跨服务调用、数据管道、架构设计、多条 Story 共享的技术方案 → 放在第一个关联 Story 下
- **服务级 Tech**：具体 API 实现、数据库操作、业务逻辑、单一模块内的技术实现 → 放在对应的 Story 下
- **无需 Tech**：纯 UI/交互类 Story、无技术复杂度的简单操作 → 标注"不涉及"

**Tech 必填内容（参照 IPD 模板 §5）：**

**【技术故事】** TL & 模块负责人
- 技术需求描述
- 关联的系统需求编号

**【A/C 验收条件】**

**功能性验收条件：**
1. 业务规则与边界
2. 系统功能约束
3. 实例化场景（正常场景 + 异常场景）

**非功能性验收条件：**
- 性能分析（接口响应指标、并发数、吞吐量）
- 可靠性（SLA、故障恢复、容灾）
- 可维护性（日志、监控、扩缩容）
- 开放性（对外接口/OpenAPI）
- 安全性（安全需求识别检视）
- 其他（兼容性、可测试性、可扩展性）

**【依赖与风险】** TL & 模块负责人
- 需求关联性、内部功能依赖、外部跨团队协作
- 技术风险、测试风险

**【技术思路】** TL & 模块负责人
- 技术实现方案
- 架构设计说明

### 步骤 3c：独立评审（SDD 风格质量门控）

创建 subagent 读取 `skills/generate-ipd-story/agents/ipd-reviewer-prompt.md`，对完整的 IPD 文档进行独立评审。

**评审维度：**

| 维度 | 满分 | 说明 |
|------|------|------|
| E/F/S/T 结构合理性 | 20 | Epic-Feature-Story-Tech 层级划分是否正确；系统级/服务级 Tech 拆分是否合理；每个服务级 Tech 是否挂载在系统级 Tech 下（禁止孤立服务级）；目录/文件命名是否符合 `Epic{N}-` / `Feature{N}.{N}-` / `Story{N}.{N}.{N}-` / `Tech-系统级-` / `Tech-服务级-` 规范格式 |
| Story 场景完整性 | 25 | 每个 Story 是否包含正常+异常+安全场景 |
| DFX 覆盖完整性 | 15 | 非功能需求是否从 ch04 正确映射 |
| QS 质量基线达标 | 20 | QS-001~008 是否全部满足/明确裁剪 |
| REQ 映射准确性 | 10 | REQ 到 Story 的映射是否无遗漏、无范围蔓延 |
| Tech 完整性 | 10 | Tech 是否覆盖了关键技术决策，系统级/服务级拆分合理 |

**评分规则：**
- 总分 ≥ 85 且无 🔴 阻塞问题 → **通过**，进入步骤 3d
- 总分 70-84 或无 🔴 问题 → **警告**，自动修复后可选重评
- 总分 < 70 或存在 🔴 问题 → **不通过**，进入步骤 3d 修复轮

### 步骤 3d：评审修复轮（最多 2 轮）

**步骤 3d.1** — 读取评审报告中的问题列表（Critical + Error 级别）
**步骤 3d.2** — 按问题列表逐项修复 IPD 文档
**步骤 3d.3** — 重新执行步骤 3c 评审（当前轮次 +1）

- 若第 2 轮评审仍不通过：输出摘要给用户，提示人工介入
- 若第 2 轮评审通过：继续步骤 4

### 步骤 4：写入输出文件

输出目录：`docs/agent-rules/2-system-requirements/output/ipd-story/YYYY-MM-DD-<project>/`

- `YYYY-MM-DD` = 今天的日期
- `<project>` = 来源文档中的项目/模块名称

若目录不存在则自动创建。完成后将输出目录路径返回给调用方。

#### 输出目录结构

```
ipd-story/YYYY-MM-DD-<project>/
├── index.md                                    # 全量索引：Epic→Feature→Story→Tech 映射表
│
├── Epic{N}-{名称}/                              # Epic = 目录 + README.md
│   ├── README.md                               #   Epic 完整内容（8 节）
│   │
│   └── Feature{N}.{N}-{名称}/                   # Feature = 目录 + README.md
│       ├── README.md                           #   Feature 完整内容（6 节）
│       │
│       └── Story{N}.{N}.{N}-{名称}/             # Story = 目录 + README.md
│           ├── README.md                       #   Story 完整内容
│           │
│           └── Tech-系统级-{名称}/               # 系统级 Tech = 目录 + README.md
│               ├── README.md                   #   系统级 Tech 完整内容
│               │                               #   （跨服务/架构/数据管道）
│               ├── Tech-服务级-{名称}.md         #   服务级 Tech = 文件
│               └── Tech-服务级-{name}.md        #   （API/DB/业务逻辑）
```

#### 命名规则

| 层级 | 格式 | 类型 | 示例 |
|------|------|------|------|
| Epic | `Epic{N}-{名称}/` | 目录 | `Epic1-日志审计/` |
| Feature | `Feature{N}.{N}-{名称}/` | 目录 | `Feature1.3-审计告警策略/` |
| Story | `Story{N}.{N}.{N}-{名称}/` | 目录 | `Story1.5.1-告警规则管理/` |
| 系统级 Tech | `Tech-系统级-{名称}/` | 目录 | `Tech-系统级-告警事件总线/` |
| 服务级 Tech | `Tech-服务级-{名称}.md` | 文件 | `Tech-服务级-告警规则CRUD接口.md` |

- `{N}` = 从 1 开始的序号
- `{名称}` = 中文名称，直接表达该层级的内容意图
- 全部层级统一使用中文命名

#### 核心规则

1. Epic / Feature / Story 三级都是**目录 + README.md**
2. 系统级 Tech 是**目录 + README.md**（跨服务协调层）
3. 服务级 Tech 是系统级目录下的**平铺 .md 文件**（具体实现层）
4. 即使 Story 没有系统级 Tech，服务级 Tech 也必须归到一个系统级目录下（代表架构边界）

#### 各层 README.md 内容

每个 README.md 写入对应层级的模板内容：

| 层级 | 模板来源 | 写入内容 |
|------|---------|---------|
| Epic README.md | 模板 §2 Epic 模板 | 8 个必填节（关键价值点、价值实现程度、用户需求设计结论、客户验证结论、质量需求、包含 Feature、跨团队协作、责任人） |
| Feature README.md | 模板 §3 Feature 模板 | 6 个必填节（关联关键价值点、特性需求分类、功能级 Playbook、特性验收条件、非功能需求约束 DFX、包含 Story） |
| Story README.md | 模板 §4 Story 模板 | 完整 Story 内容（用户故事、场景说明、A/C 验收条件、操作步骤、期望效果、依赖与风险、技术思路、工作量评估、责任人） |
| 系统级 Tech README.md | 模板 §5 Tech 模板 | 完整 Tech 内容（技术故事、A/C 验收条件、依赖与风险、技术思路） |
| 服务级 Tech .md | 模板 §5 Tech 模板 | 完整 Tech 内容（同上） |

```markdown
# IPD Story 索引 — <项目名称>

> 生成时间: YYYY-MM-DD HH:mm
> 来源: docs/agent-rules/2-system-requirements/output/<sysreq_dir>/

## 结构概览

| 层级 | 数量 |
|------|------|
| Epic | N |
| Feature | N |
| Story | N |
| Tech-系统级 | N |
| Tech-服务级 | N |

## 完整映射

  - **Feature1.1-xxx** → [README.md](Epic1-xxx/Feature1.1-xxx/README.md)
    - **Story1.1.1-xxx** → [README.md](...)
      - Tech-系统级-xxx → [README.md](...)
      - Tech-系统级-xxx → [README.md](...)
        - Tech-服务级-xxx.md
        - Tech-服务级-xxx.md
```

## 生成策略对比

| 阶段 | 内容 | 确认方式 |
|------|------|---------|
| Step 2 结构规划 | E/F/S/T 树形结构 | 用户确认或后台跳过 |
| Step 3a 框架 | 字段骨架（标题+核心字段） | 用户确认或后台跳过 |
