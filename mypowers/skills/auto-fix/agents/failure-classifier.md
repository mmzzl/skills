# auto-fix 失败用例综合分类器

对测试失败用例进行多维信息综合分类。

## 参数

| 参数 | 说明 |
|------|------|
| `${task_dir}` | 任务目录名 |
| `${caseId}` | 用例 ID（RF: 测试名称; MCP: caseCode） |
| `${framework}` | 框架类型：`rf` 或 `mcp` |

---

对 `${framework}` 框架下的失败用例 `${caseId}` 进行根因分析与分类。请依次执行：

## 1. 读取测试执行证据

### RF 框架

读取 `.cospowers/auto-test/tasks/${task_dir}/failed_tests.json`，找到 `name === "${caseId}"` 或 `id === "${caseId}"` 的条目。

获取：name、status、message、tags、suite、suiteSource、keywords[]（关键字调用链）。

分析关键字调用链：
- 找到第一个 `status === "FAIL"` 的关键字
- 提取 name、library、args、messages
- 检查失败关键字之前的异常迹象（warning、异常返回值）
- 检查失败后是否有关键字被阻断（NOT RUN）

### MCP/TP 框架

读取 `.cospowers/auto-test/tasks/${task_dir}/${caseId}_detail.json`。

获取：caseCode、caseName、caseStatus、caseSteps[]（步骤列表）。

对每个步骤：
- `stepIndex`、`stepType`（execute/check）
- `stepContent` — 步骤描述
- `actions[]` — 操作列表
  - `actionDesc`、`actionStatus`（success/fail）
  - `actionResult` — 执行结果详情
  - `toolName`、`toolParam` — 调用的 MCP 工具信息

读取截图：`.cospowers/auto-test/tasks/${task_dir}/analysis/screenshots/${caseId}/` 下的截图文件，进行视觉分析。

读取 MCP 日志：`.cospowers/auto-test/tasks/${task_dir}/mcp_logs_${caseId}.log`，重点关注：
- 工具接收入参（判断是否传参问题）
- 工具内部异常/错误堆栈（判断是否工具内部逻辑问题）
- 返回结果

### 假通过检测（仅 MCP/TP）

对每个 `stepType === "check"` 且 `actionStatus === "success"` 的步骤：
- 读取截图，检查页面标题/导航位置是否与步骤描述一致
- 检查搜索框条件是否正确
- 检查数据量是否合理

若发现假通过，在报告中标注：实际页面 vs 预期页面、对后续步骤的影响链。

## 2. 收集辅助证据

### 2a. 产品运行日志（必有）

**每个失败用例必须读取**产品运行时日志。日志由其他技能自动获取并存放于 `.cospowers/auto-test/tasks/${task_dir}/logs/` 目录下，分类器直接读取该目录即可，无需自行拉取。

如果 `logs/` 目录为空或不存在，在分类报告中标注"日志缺失"，置信度自动降级为低。

日志分析要点：
- 有异常堆栈/panic/error → 倾向 **B 类**
- 有连接超时/服务不可用/资源不足 → 倾向 **C 类**
- 请求正常处理但返回值不符合预期 → 倾向 **B 类**
- 日志中无任何异常 → 可能是 **A 类**（测试期望值写错）或 **B 类**（逻辑错误但无日志）
- 无法获取日志 → 标注"日志缺失"，置信度自动降级为低

### 2b. 设计方案

**何时获取**：需要判断测试断言期望值是否合理时（区分 A 类 vs B 类）。

查找项目中的设计文档：
- `doc/design/`、`docs/design/`、`design/` 目录
- API 定义文件（OpenAPI yaml、Proto 文件）
- 通过 `daedalus-knowledge` 检索相关技术方案

设计文档分析要点：
- 接口预期返回 → 测试期望值是否与设计一致？
- 预期值一致但实际返回错误 → **B 类**
- 预期值与设计不符 → **A 类**
- 找不到设计文档 → 标注"设计文档缺失"，依赖其他证据

### 2c. 源代码

**何时获取**：初步判定为 B 类时，确认缺陷位置。

根据失败关键字/API路径搜索相关代码：
- 定位相关函数/方法
- 检查代码逻辑是否与设计方案一致
- 确认 Bug 具体位置

## 3. 根因分类（A/B/C/D）

综合以上多维证据进行分类：

### A 类 — 用例编写错误
- 元素定位器错误或不准确（Element not found、locator 不匹配）且日志显示服务正常
- 关键字参数传递错误（参数类型不匹配、缺少必填参数）
- 等待/同步缺失（元素未加载就操作）
- BuiltIn 断言失败且期望值写错（与设计文档/API定义不一致）
- MCP 工具传参错误：步骤描述中的参数不正确
- 硬编码值不合理

### B 类 — 业务代码缺陷
- **日志中有异常堆栈、panic、error** + 接口返回异常
- 接口返回业务错误码或异常数据，**日志中无环境异常**
- 页面渲染异常（白屏、JS 错误）
- 后端逻辑 bug 导致数据错乱
- 权限/认证逻辑缺陷
- BuiltIn 断言失败且实际值不对（期望值与设计一致，但实际返回错误）
- **设计文档明确预期行为，但代码实现不符**

### C 类 — 环境/数据问题
- 测试环境不可用（连接超时、502/503）
- **日志显示服务未启动、资源不足、依赖服务异常**
- 测试数据被污染（依赖数据不存在或状态不对）
- 网络超时、DNS 解析失败
- 容器不存在、服务未启动

### D 类 — MCP 工具问题（仅 MCP/TP 框架）
- 工具收到的入参**正确**（与步骤描述一致），但工具内部执行异常
- **MCP 日志中有工具内部错误/异常堆栈**
- 工具依赖的服务异常（数据库、缓存等）
- 重置模板/下发流量等工具自身处理失败

### 分类判断优先级（多维证据综合）
1. **日志中有异常堆栈/panic + 接口返回异常** → B 类（高置信度）
2. 错误信息包含 timeout/502/503/connection refused + **日志中服务未启动** → C 类
3. MCP 工具调用且入参正确但内部异常 + **MCP日志中有工具内部错误** → D 类
4. MCP 工具调用且入参错误 → A 类（步骤描述中的参数问题）
5. **设计文档明确预期行为 + 测试断言与设计一致 + 实际结果不对** → B 类
6. **设计文档未明确定义该行为 + 测试断言基于历史版本行为 + 实际返回与历史版本不一致** → **优先归为 B 类**（代码行为变更），除非能明确证明测试期望值本身就是错误的。见下方"A 类排除规则"
7. **设计文档明确预期行为 + 测试断言与设计不一致** → A 类（需满足 A 类高置信度门槛）
8. 元素定位/操作失败且非环境原因 → 检查日志判断 A 还是 B
9. 关键字执行通过但返回值不符合预期 → 检查设计文档判断 A 还是 B

### A 类排除规则

A 类判定需要更高的证据标准——**不能仅凭"实际结果与测试期望不符"就归为 A 类**。归为 A 类需同时满足以下至少两项：

1. **设计文档明确支持测试的期望值**（不是"没有设计文档"，而是设计文档**明确说**期望值应该是 X）
2. **产品代码逻辑审查确认代码行为符合设计意图**（代码不是 bug，是设计如此）
3. **测试用例使用的断言值可以被独立验证为不合理**（如硬编码了不可能的值、参数格式错误等）

仅凭"测试期望值与实际不符"归为 A 类，置信度必须降级为**中或低**，并在报告中标注"分类不确定，建议人工复核"。

### 分类不确定性规则

当多维证据不足以在 A 类和 B 类之间做出高置信度判定时（置信度 < 70%）：

- **优先归为 B 类**（代码问题），而非 A 类（用例问题）
- **Why**: B 类误判为 A 类的后果更严重——代码 bug 被掩盖，测试用例被错误修改以"适配"bug，导致回归测试失效。A 类误判为 B 类的后果较轻——会触发代码审查，审查时可发现实际是用例问题并修正方向
- 在分类报告中标注"分类不确定，建议人工复核"

## 4. 置信度评估

- **高（>80%）**：多维证据（测试+日志/设计）交叉验证，结论明确
- **中（50-80%）**：可推断原因，但部分证据缺失（如无日志、无设计文档）
- **低（<50%）**：仅有测试执行结果，缺少辅助证据，多种原因均有可能

**A 类特殊规则**：归为 A 类但未满足 A 类排除规则中至少两项的，置信度上限为中。

## 5. 保存分类报告

写入 `.cospowers/auto-test/tasks/${task_dir}/analysis/${caseId}.md`：

### RF 框架报告格式

```
# ${testName} 失败分类报告

## 基本信息
- **测试名称**: ${testName}
- **测试 ID**: ${id}
- **执行状态**: FAIL
- **所属套件**: ${suite}
- **用例文件**: ${suiteSource}
- **标签**: ${tags}
- **执行时长**: ${elapsed}ms

## 证据来源

| 证据类型 | 状态 | 摘要 |
|:---|:---|:---|
| 测试执行结果 | ✅ 已获取 | ${失败关键字 + 错误信息摘要} |
| 产品运行日志 | ✅/⚠️/❌ | ${日志关键发现 或 "未获取到"} |
| 设计方案 | ✅/⚠️/❌ | ${设计文档关键发现 或 "未找到相关设计文档"} |
| 源代码 | ✅/⚠️/❌ | ${代码分析关键发现 或 "分类阶段未读取"} |

## 结论

**根因分类**: ${A类/B类/C类}
**置信度**: ${高/中/低}（${百分比}）
**一句话结论**: ${一句话概括}

## 失败定位

**失败关键字**: ${keyword.name}（${keyword.library}）
**关键字参数**: `${keyword.args}`
**错误信息**:
\`\`\`
${关键错误内容}
\`\`\`

**关键字调用链**（失败点前后各 3 个）：

| 序号 | 关键字 | 库 | 状态 | 参数 | 耗时 |
|:---|:---|:---|:---|:---|:---|
| N-2 | ... | ... | PASS | ... | ...ms |
| N-1 | ... | ... | PASS | ... | ...ms |
| **N** | **${name}** | **${library}** | **FAIL** | **${args}** | **${elapsed}ms** |
| N+1 | ... | ... | NOT RUN | — | — |

## 详细分析

### 测试证据
${测试结果分析}

### 日志证据
${日志分析结论，或说明未获取原因}

### 设计证据
${设计文档对照结论，或说明未找到原因}

### 综合判定
${多维证据交叉验证结论}

## 处置建议

**建议操作**: ${修改用例 / 修复代码 / 排查环境}
**具体步骤**:
1. ...
2. ...
```

## 6. 返回 JSON 摘要

### RF 框架

```json
{
  "framework": "rf",
  "testName": "${testName}",
  "testId": "${id}",
  "suite": "${suite}",
  "rootCause": "A|B|C",
  "confidence": "${高|中|低}(${百分比})",
  "conclusion": "${一句话结论}",
  "failedKeyword": "${失败关键字名称}",
  "evidenceSources": {
    "testResult": true,
    "logs": true|false,
    "designDoc": true|false,
    "sourceCode": true|false
  },
  "reportPath": "${task_dir}/analysis/${caseId}.md"
}
```

### MCP/TP 框架

```json
{
  "framework": "mcp",
```
