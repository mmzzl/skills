# TR1 产品规格基线（Product Spec Baseline）

> **[AI 读取引导]**
> 本文件用于承接 TR1 用户需求阶段的产品规格基线要求，与 DFX checklist 分工如下：
>
> **使用方式**：
> - 填写用户需求模板 §2.1 产品规格汇总 时，对照本文件逐项自检
> - AI 评审时，检查被评审文档是否覆盖本文件要求，并确认已输出产品规格变化说明自检结果
> - 本文件关注产品规格变化说明、硬件平台边界、授权要求确认；DFX 能力指标请改查 `user-requirement-dfx-baseline-checklist.md`

## 高密度摘要

**本文件的目标**：把 TR1 用户需求中的产品规格基线从 DFX checklist 中独立出来，作为 §2.1 产品规格汇总的填写依据和评审标准。

**全局强约束**：
- **必须**：相对上版本有变化（升高 / 降低 / 裁剪）的规格项，必须说明变化原因
- **必须**：支持的硬件平台须明确列出；不支持的硬件平台须说明边界或原因
- **必须**：授权模式设计和授权对接基线活动须明确记录确认结果
- **必须**：主检查粒度到“维度 + 大类”；子类和具体场景说明回查《产品规格基线（需求）》源表
- **禁止**：只写“规格已评估”“规格无明显变化”等模糊结论，不附规格变化说明自检结果

**与 QS-004 的关系**：
- 产品规格基线自检是 `RULE-UR-QS-004` 的组成部分之一
- TR1 通过 QS-004 时，必须同时具备：
  1. 产品规格变化说明自检结果
  2. DFX Checklist 自检结果

## 第二部分：产品规格基线要求

### RULE-DFX-SPEC-001: 产品规格基线全量自检

- **等级**：L0（底线，必须达成）
- **适用范围**：所有提交 TR1 评审的产品版本
- **基线要求**：
  - 对照产品规格基线覆盖范围表，按适用维度和大类填写本产品规格参数
  - 相对上版本有变化（升高或降低）的规格项须注明变化原因
  - 支持的硬件平台须明确列出，不支持的须说明原因
  - 授权模式设计须知会数字化平台授权中心
  - 授权对接基线活动确认结果须在文档中可查
- **检查方式**：对照 §2.1 产品规格汇总中的规格表、自检表和证据链接，按适用大类逐项核查

### 规格基线覆盖说明

产品规格基线源表覆盖约 800+ 条标准化条目（具体数量随版本迭代更新）。本 checklist 在主检查粒度上按“维度 + 大类”组织，不下钻到子类；如需核对具体条目，请回查《产品规格基线（需求）》源表中的对应子类和具体场景说明。

## 第三部分：产品规格变化说明自检表

> 填写用户需求文档 §2.1 产品规格汇总时，按以下表格逐项核对。若文档采用外部附件承载，也需在文档内给出链接或定位信息。

| 自检项 | 检查要求 | 自检结果 | 说明 / 证据链接 |
|---|---|---|---|
| 规格维度与大类完整 | 已按适用范围填写规格维度与大类，且能覆盖本版本需声明的关键规格项 | 通过/不通过 | *[章节位置或链接]* |
| 上版本值与本版本目标值完整 | 每个需比较的规格项均填写上版本值和本版本目标值；如无上版本，也要说明原因 | 通过/不通过 | *[章节位置或链接]* |
| 变化原因明确 | 对升高、降低、裁剪、不变但需说明的规格项，已写明变化原因或保持原因 | 通过/不通过 | *[章节位置或链接]* |
| 硬件平台边界明确 | 已列出支持的硬件平台；存在不支持平台时，已说明边界、限制或原因 | 通过/不通过 | *[章节位置或链接]* |
| 授权模式设计明确 | 已说明授权模式设计方案，并标明是否需数字化平台授权中心参与评审 | 通过/不通过 | *[章节位置或链接]* |
| 授权对接基线活动已确认 | 已记录授权对接基线活动确认结果，缺项时有待补充说明 | 通过/不通过 | *[章节位置或链接]* |
| 规格变化说明可追溯 | 文档内可定位到产品规格变化说明、自检结论、适用维度/大类及相关附件 / 链接 | 通过/不通过 | *[章节位置或链接]* |
