# 总体设计阶段 · 背景情报简报

> **AI 使用说明**
>
> 本文档由 AI 在开始生成总体设计前发送给用户。
> 目的是收集 AI **无法仅从系统需求文档和代码库中自行推断**的背景情报、架构约束、团队决策，以及技术需求设计所需的前置输入。
>
> **填写规则：自由格式即可。**
> 重点提供：已有架构图、历史设计文档、团队已定下的技术选型、不能动的现有服务，以及与技术需求设计相关的已有输入材料。
> AI 会基于这些约束输入提出 2-3 个架构方案让你选择，而不是凭空设计。
>
> **只需要填你知道的部分，不知道的跳过。**
> AI 校验的最低标准是：§0.1 本次设计类型需明确；§1 现有架构与系统资料 或 §2 依赖文档 至少有一项有内容。

## §1 现有架构与系统资料 ⭐强烈建议提供

> **这是最有价值的输入。** AI 看到现有架构资料后可以：
> - 准确识别哪些模块已经存在，避免重复设计
> - 定位到知识库中对应模块的接口和数据结构
> - 理解现有服务的边界，设计更准确的集成方案
>
> 可提供当前系统已有的架构图、部署图、模块图、流程图、接口图、运行链路图等资料。
> 若没有现成图纸，可用文字说明当前系统的主要组成、交互关系、部署方式和关键业务流程。
>
> **支持格式（任选其一即可）：**
> - draw.io 文件路径（.drawio 或 .xml）
> - Mermaid/PlantUML 代码（直接粘贴）
> - **文字描述**：列出各服务名称及调用/依赖关系，AI 会据此生成架构图
>
> **建议覆盖的资料类型（有多少提供多少）：**
> - 场景/用例资料：核心业务流程、关键用户场景、典型操作路径
> - 逻辑架构资料：系统模块、服务划分、职责边界、主要数据对象
> - 开发结构资料：代码仓库、工程结构、组件依赖、技术栈
> - 运行机制资料：进程/线程模型、任务调度、消息通信、关键链路
> - 物理部署资料：服务器、设备、网络拓扑、部署环境、硬件连接
> - 接口与集成资料：外部系统、协议、API、数据交换方式
> - 历史设计资料：已有设计文档、评审纪要、架构决策记录
>
> **文字描述示例：**
> ```
> 服务列表及在本项目中承担的功能：
> - API Gateway：本项目所有请求的统一入口，负责路由分发和鉴权
> - User Service：承担本项目的用户模块，处理注册、登录和权限校验
> - Order Service：承担本项目的订单模块，覆盖从下单到履约的完整流程
> - Payment Service：承担本项目的支付模块，处理扣款、退款和对账
> - MySQL：存储本项目的订单和用户数据
> - Redis：为用户模块提供登录态缓存
>
> 调用关系：
> - 客户端 → API Gateway（HTTP）
> - API Gateway → User Service / Order Service（gRPC）
> - Order Service → Payment Service（gRPC）
> - Order Service → MySQL（读写），User Service → Redis（缓存）
> ```

*(在这里贴文件路径、代码，或用文字描述现有系统资料与服务关系...)*
